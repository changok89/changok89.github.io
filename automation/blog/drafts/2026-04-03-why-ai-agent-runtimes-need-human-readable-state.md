# AI 에이전트 런타임은 왜 사람에게 읽히는 상태를 남겨야 하는가

AI 에이전트를 잠깐 데모하는 단계에서는 잘 안 보이지만, 실제로 며칠씩 돌리거나 여러 작업을 병렬로 맡기기 시작하면 금방 중요한 질문이 생긴다.

- 지금 이 에이전트는 무엇을 하고 있지?
- 멈춘 건가, 기다리는 건가?
- 실패했다면 어디서 실패했지?
- 내가 지금 개입해야 하나, 조금 더 기다려야 하나?

처음에는 모델 성능이나 프롬프트가 가장 중요해 보인다. 물론 그것도 중요하다. 하지만 운영 단계로 가면 체감상 더 자주 발목을 잡는 건 **상태 가시성 부족**이다. 에이전트가 똑똑하더라도, 사람이 현재 상태를 읽을 수 없으면 시스템은 금방 답답해진다.

내가 보기엔 장기 운영에서 좋은 에이전트 런타임은 단순히 일을 잘 처리하는 런타임이 아니다. **사람이 읽고 개입할 수 있을 정도로 상태를 남기는 런타임**이어야 한다.

## 왜 "상태"가 성능만큼 중요해지는가

사람이 직접 작업하면 머릿속에서 상태를 추적할 수 있다. 지금 문서를 읽는 중인지, 테스트가 도는 중인지, 막혀서 조사 중인지 스스로 알고 있다. 하지만 에이전트 작업은 바깥에서 보면 종종 검은 상자처럼 보인다.

예를 들어 아래 세 상황은 겉보기에는 모두 "응답이 없다"로 보일 수 있다.

1. 에이전트가 대용량 저장소를 읽는 중
2. 테스트가 오래 걸려서 대기 중
3. 외부 승인을 기다리는 중
4. 이미 실패했는데 표시가 없음

사람 입장에서는 이 넷이 전혀 다르다.

- 1번이면 기다리면 된다
- 2번이면 진행률이나 예상 시간을 보면 된다
- 3번이면 내가 승인해야 한다
- 4번이면 재시도나 진단이 필요하다

그런데 시스템이 이 차이를 보여주지 않으면 운영자는 모든 무응답을 같은 것으로 취급하게 된다. 그래서 쓸데없이 재시작하고, 이미 진행 중인 작업에 중복 지시를 보내고, 승인 대기를 놓치고, 실패 원인을 늦게 발견하게 된다.

즉, 상태를 잘 보여주는 건 보기 좋은 부가 기능이 아니라 **운영 비용을 줄이는 핵심 기능**에 가깝다.

## 로그만 많다고 상태가 보이는 건 아니다

여기서 흔히 생기는 오해가 있다. 로그가 많으면 가시성이 좋아졌다고 생각하는 것이다. 실제로는 그 반대인 경우도 많다.

터미널 로그가 길게 쏟아진다고 해서 사람이 현재 상태를 곧바로 이해할 수 있는 건 아니다. 오히려 아래 질문에는 잘 답하지 못하는 경우가 많다.

- 지금 단계는 무엇인가
- 마지막으로 성공한 단계는 무엇인가
- 현재 대기 원인은 무엇인가
- 사용자 입력이 필요한가
- 자동 재시도가 가능한 실패인가
- 지금 중단해도 안전한가

이런 질문에 답하려면 원시 로그 외에 **요약된 상태 표현**이 필요하다.

예를 들면 이런 식이다.

- 단계: 저장소 탐색 중
- 진행: 관련 파일 12/40 확인
- 대기 사유: 사용자 승인 필요 (`rm` 포함 명령)
- 마지막 성공 단계: 테스트 환경 준비 완료
- 다음 액션: 승인 후 통합 테스트 실행

이 정도만 있어도 운영 경험이 완전히 달라진다.

## 사람이 읽는 상태는 왜 꼭 인간 친화적이어야 하나

기계가 읽기 쉬운 상태와 사람이 읽기 쉬운 상태는 다르다. 둘 다 필요하지만, 운영 중간 개입을 생각하면 인간 친화적인 표현이 꼭 있어야 한다.

예를 들어 내부적으로는 이런 구조가 있을 수 있다.

- `phase=tool_call`
- `subphase=exec_approval_pending`
- `retryable=false`
- `blocked_on=user`

이건 시스템끼리 주고받기에는 좋다. 하지만 사람이 바로 읽고 판단하기엔 불친절하다.

사람에게는 차라리 이렇게 보여주는 편이 낫다.

- 현재 상태: 위험 명령 승인을 기다리는 중
- 왜 멈췄나: 삭제성 명령이 포함되어 자동 진행이 중단됨
- 지금 필요한 일: 승인 또는 명령 수정
- 기다려도 되나: 아니오. 사용자 입력이 필요함

운영 중에는 결국 사람이 빠르게 판단해야 한다. 그러니 상태도 **판단 가능한 문장**이어야 한다.

## 장기 실행 작업일수록 상태 설계가 중요하다

짧은 단발성 작업은 대충 넘어갈 수 있다. 몇 초 안에 끝나면 상태 표현이 다소 거칠어도 큰 문제가 없다. 하지만 아래 같은 작업은 얘기가 다르다.

- 큰 저장소 리팩터링
- 여러 PR 생성/검토 자동화
- 모바일 빌드와 테스트 반복
- 장시간 돌아가는 감시/운영 점검 작업
- 다단계 승인이 포함된 자동화

이런 작업에서는 상태가 없으면 불안이 커진다. 작업이 실제로 진행 중인지, 조용히 죽었는지, 다음에 무슨 일이 일어날지 알 수 없기 때문이다.

특히 여러 세션이나 서브에이전트를 함께 돌릴 때는 상태가 더 중요하다.

- 어떤 작업은 아직 탐색 중인지
- 어떤 작업은 사용자 응답을 기다리는지
- 어떤 작업은 거의 끝났는지
- 어떤 작업이 비정상적으로 오래 걸리는지

이걸 한눈에 볼 수 있어야 병렬 운영이 가능하다. 그렇지 않으면 여러 에이전트를 돌린다는 장점이 오히려 혼란으로 바뀐다.

## 좋은 상태 표현은 무엇을 포함해야 하나

내 기준으로 최소한 아래 정도는 있어야 사람이 개입하기 쉽다.

### 1. 현재 단계

지금 무엇을 하는 중인지 보여줘야 한다.

- 문서 읽는 중
- 코드 수정 중
- 테스트 실행 중
- 외부 승인 대기 중
- 사용자 답변 작성 중

이게 없으면 무응답과 작업 중을 구분하기 어렵다.

### 2. 대기 이유

멈춘 것처럼 보일 때 특히 중요하다.

- 장시간 테스트 실행 중
- 네트워크 응답 대기 중
- 사용자 승인 필요
- 브라우저 로그인 필요
- 재시도 백오프 중

"왜 기다리는지"를 모르면 사람은 불필요하게 개입한다.

### 3. 마지막으로 성공한 단계

에러가 나거나 멈췄을 때 복구 기준점이 된다.

- 저장소 읽기 완료
- 초안 생성 완료
- 테스트 환경 준비 완료
- 이슈 목록 수집 완료

이 정보가 있으면 어디서부터 다시 시작할지 판단하기 쉽다.

### 4. 다음 예상 액션

현재 상태뿐 아니라 곧 이어질 동작도 보여주면 훨씬 안심된다.

- 승인 후 테스트 재개 예정
- 현재 테스트 완료 후 결과 요약 예정
- 재시도 횟수 초과 시 실패로 전환 예정

사람은 미래를 조금이라도 예측할 수 있을 때 시스템을 더 신뢰한다.

### 5. 사용자 개입 필요 여부

이건 따로 강조하는 편이 좋다.

- 개입 필요 없음
- 승인 필요
- 누락된 정보 필요
- 로그인/권한 확인 필요

개입 필요 여부가 명확하면 알림 설계도 쉬워진다.

## 상태를 잘 남기면 디버깅보다 운영이 먼저 쉬워진다

상태 가시성 이야기를 하면 보통 디버깅을 먼저 떠올린다. 맞는 말이다. 하지만 내가 체감하기엔 디버깅 이전에 **운영이 먼저 편해진다**.

예를 들면,

- 사용자는 괜히 "아직이야?"를 덜 묻게 된다
- 운영자는 어디에 개입해야 하는지 더 빨리 안다
- 승인 대기와 실제 실패를 구분할 수 있다
- 장시간 작업을 더 편하게 맡길 수 있다
- 중복 지시와 중복 실행이 줄어든다

결국 시스템 피로도가 낮아진다. 이건 생각보다 중요하다. 상태가 불투명한 시스템은 기술적으로 돌아가도 사람이 점점 덜 쓰게 된다.

## 에이전트 런타임에서 자주 놓치는 상태 안티패턴

실무적으로는 아래 패턴이 특히 답답하다.

### 1. 모든 걸 "working"으로만 표시하는 경우

이건 사실상 상태가 없는 것과 비슷하다. 사람은 무엇을 하고 있는지, 정상 진행인지, 막혔는지 알 수 없다.

### 2. 실패했는데 마지막 로그만 남고 상태 전환이 없는 경우

중요한 건 로그 한 줄보다 "실패 상태로 전환되었는가"다. 실패가 구조적으로 표시되지 않으면 모니터링도 어려워진다.

### 3. 승인 대기를 일반 대기와 구분하지 않는 경우

이건 운영 경험을 크게 망친다. 사용자 액션이 필요한데도 그냥 시간이 오래 걸리는 작업처럼 보이기 때문이다.

### 4. 상태는 세밀한데 요약이 없는 경우

세부 이벤트는 풍부한데, 정작 요약 문장이 없어서 사람이 전체 상황을 이해하기 어렵다. 내부 이벤트 스트림과 사용자용 상태 요약은 따로 설계하는 편이 낫다.

### 5. 완료 보고에 불확실성이 빠지는 경우

"완료"라고만 적혀 있는데 실제로는 일부 경로를 확인하지 못했을 수도 있다. 완료 상태에도 확신 수준이나 미확인 범위를 같이 남기는 편이 안전하다.

## 내가 선호하는 최소 상태 모델

거창한 시스템이 아니어도 아래 정도만 있어도 많이 좋아진다.

- 상태: running / blocked / failed / completed
- 단계: read / plan / edit / test / review / report
- 대기 사유: approval / input / network / retry / login
- 마지막 성공 단계
- 다음 액션
- 사용자 개입 필요 여부

이걸 내부적으로 구조화해 두고, 사용자에게는 문장으로 풀어 보여주면 된다.

예를 들어 이렇게 말이다.

> 상태: blocked  
> 단계: test  
> 이유: 삭제성 명령 승인을 기다리는 중  
> 마지막 성공: 코드 수정 완료  
> 다음 액션: 승인 후 테스트 실행  
> 사용자 개입: 필요

이 정도면 사람이 어디서 뭘 해야 하는지 거의 바로 안다.

## 결국 상태는 신뢰의 인터페이스다

에이전트를 쓰는 사람은 모델 내부를 알 수 없다. 그래서 신뢰는 결과 품질만으로 생기지 않는다. **중간 상태가 얼마나 읽히는가**에서도 크게 생긴다.

- 무슨 일을 하고 있는지 보이고
- 왜 기다리는지 설명되고
- 실패가 숨겨지지 않고
- 내가 언제 개입해야 하는지 알려주면

사람은 시스템을 훨씬 덜 불안하게 느낀다.

나는 앞으로 에이전트 런타임을 평가할 때 모델 능력만큼이나 이 질문이 중요해질 거라고 본다.

> 이 런타임은 사람에게 현재 상태를 얼마나 잘 설명하는가?

에이전트가 더 자율적으로 움직일수록, 사람은 역설적으로 더 좋은 상태 표현을 필요로 한다. 결국 운영 가능한 시스템은 조용히 똑똑한 시스템이 아니라, **필요한 순간에 자기 상태를 사람 언어로 설명할 수 있는 시스템**에 더 가깝다.
