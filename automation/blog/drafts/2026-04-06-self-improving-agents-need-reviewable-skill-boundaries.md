최근 GeekNews 새 글 목록에서 `Hermes Agent`처럼 **경험으로부터 스킬을 생성·개선하는 자기 학습형 에이전트**라는 표현이 눈에 들어왔다. 이런 설명은 꽤 강하게 끌린다. 사람 입장에서는 당연히 기대하게 된다.

- 반복할수록 더 똑똑해진다
- 자주 하는 일을 스스로 익힌다
- 사람이 일일이 규칙을 써주지 않아도 된다
- 운영할수록 자동화 품질이 올라간다

이상적으로는 정말 좋아 보인다. 나도 이런 방향 자체는 맞다고 생각한다. 에이전트가 매번 처음부터 배우는 것보다, 경험에서 패턴을 축적하는 쪽이 장기적으로 더 강력할 가능성이 크다.

그런데 실무에서 바로 떠오르는 질문은 조금 다르다.

**그 스킬을 누가 검토하고, 어디까지 믿고, 문제가 생기면 어떻게 되돌릴 건가?**

내가 보기엔 자기개선형 에이전트의 핵심 문제는 “스스로 발전하느냐”가 아니다. 그보다 먼저 **리뷰 가능한 스킬 경계가 있느냐**가 훨씬 중요하다.

## 왜 자기개선형 에이전트는 멋있어 보이면서도 바로 위험해지나

일반적인 자동화는 그래도 구조가 비교적 단순하다.

- 사람이 규칙을 만든다
- 에이전트가 그 규칙에 맞게 실행한다
- 결과를 보고 사람이 수정한다

여기서는 책임 경로가 분명하다. 규칙이 잘못되면 고칠 위치도 비교적 선명하다.

반면 자기개선형 구조는 이 흐름이 달라진다.

- 에이전트가 작업을 수행한다
- 결과를 보고 패턴을 추출한다
- 그 패턴을 새 스킬이나 규칙으로 저장한다
- 다음 실행부터 그 규칙이 다시 적용된다

이 구조가 무서운 이유는 단순하다. **실패가 한 번으로 끝나지 않고 시스템 규칙으로 굳어질 수 있기 때문**이다.

즉, 잘못된 추론이 단발성 실수에서 끝나지 않고 재사용 가능한 자동화 자산처럼 쌓일 수 있다.

## 가장 먼저 생기는 문제는 정확도보다 추적 가능성이다

자기개선형 에이전트를 이야기하면 보통 먼저 정확도를 걱정한다.

- 잘못 배운 패턴을 일반화하지 않을까
- 특정 작업에 과적합되지 않을까
- 잘못된 도구 선택을 습관화하지 않을까

이것도 맞다. 하지만 운영 관점에서는 그보다 먼저 **추적 가능성**이 더 큰 문제다.

예를 들어 어떤 작업이 이상하게 망가졌다고 해 보자. 이때 원인을 좁히려면 아래를 알아야 한다.

- 지금 적용된 스킬이 무엇인지
- 그 스킬이 언제 생성되었는지
- 어떤 실행 로그를 근거로 만들어졌는지
- 누가 승인했는지
- 이전 버전으로 되돌릴 수 있는지

이 다섯 개가 없으면, 자기개선은 거의 곧바로 블랙박스가 된다.

사람이 직접 적은 규칙은 비효율적이어도 최소한 읽을 수는 있다. 반면 자동 생성된 스킬이 계속 쌓이기 시작하면, 잘못된 결과보다 **왜 그런 결과가 나왔는지 설명하기 어려워지는 것**이 더 빨리 문제 된다.

## 스킬 자동 생성이 특히 위험한 지점

내가 보기엔 아래 세 가지에서 특히 위험도가 급격히 올라간다.

### 1. 작업 경계가 흐린 상태에서 스킬이 생성될 때

자기개선이 잘 작동하려면 먼저 작업 종류가 잘 분리돼 있어야 한다.

예를 들면 이런 식의 경계가 있어야 한다.

- 조사 스킬
- 리뷰 스킬
- 코드 수정 스킬
- 글쓰기 초안 스킬
- 운영 점검 스킬

그런데 경계가 없는 상태에서 스킬을 만들기 시작하면, 에이전트는 여러 상황을 한 덩어리 규칙으로 묶어버릴 가능성이 크다.

그러면 아래 같은 일이 생긴다.

- 조사할 때 쓰던 단축 규칙이 리뷰에도 섞인다
- 임시 우회가 장기 규칙으로 승격된다
- 특정 저장소에서만 맞는 규칙이 범용 스킬처럼 재사용된다

즉, 자기개선 이전에 **업무 분류 체계**가 먼저 필요하다.

### 2. 실패한 실행이 좋은 사례처럼 저장될 때

에이전트는 결과가 통과되었다고 해서 항상 좋은 방법을 배운 건 아니다.

예를 들어 이런 상황은 실제로 자주 일어난다.

- 우연히 테스트가 통과했다
- 사람이 바빠서 대충 승인했다
- 외부 서비스 일시 오류를 잘못된 재시도 규칙으로 학습했다
- 특정 환경에서만 맞는 workaround를 일반 규칙으로 저장했다

이건 ML 쪽의 학습 데이터 오염과 비슷하지만, 운영 자동화에서는 더 즉각적으로 위험하다. 왜냐하면 **다음 실행부터 바로 행동 규칙으로 적용**되기 때문이다.

### 3. 사람이 읽기 어려운 형태로 규칙이 누적될 때

스킬은 재사용 가능한 자산이어야 한다. 그런데 자동 생성된 스킬이 아래처럼 쌓이면 곤란하다.

- 이름이 모호하다
- 적용 조건이 불명확하다
- 입력과 출력이 명시되지 않는다
- 실패 시 중단 조건이 없다
- 예외 상황이 문장 속에만 숨어 있다

이런 스킬이 많아지면 성능이 아니라 유지보수성이 먼저 무너진다. 결국 사람은 “자동 개선된 지식”을 신뢰하지 못하고 다시 수동 운영으로 돌아가게 된다.

## 그래서 먼저 필요한 건 학습 능력이 아니라 리뷰 가능한 경계다

자기개선형 에이전트를 진짜 실전에서 쓰고 싶다면, 내가 먼저 요구할 건 모델 성능이 아니다. 아래 구조다.

### 1. 스킬 단위가 파일이나 객체로 분리되어야 한다

최소한 아래 정보는 독립적으로 보여야 한다.

- 이름
- 목적
- 적용 조건
- 입력 형식
- 출력 형식
- 실패 시 행동
- 생성 시각
- 생성 근거
- 승인 상태

이 정도는 있어야 사람도 검토할 수 있다.

### 2. 자동 생성과 자동 적용은 분리해야 한다

이건 거의 필수라고 본다.

- 자동 생성: 가능
- 자동 제안: 가능
- 자동 적용: 매우 제한적

즉, 새 스킬은 먼저 `candidate` 상태로 쌓이고, 사람이 확인하거나 별도 검증을 통과한 뒤에만 `approved`로 승격되는 편이 안전하다.

### 3. 롤백 경로가 명확해야 한다

자기개선형 시스템에서 제일 위험한 문장은 “최근에 알아서 좋아졌습니다”다. 좋아진 이유를 설명할 수 없다면, 나빠졌을 때도 되돌리기 어렵다.

그래서 최소한 아래는 필요하다.

```text
skill_id: review-pr-comments-v3
parent: review-pr-comments-v2
status: candidate
created_from_run: 2026-04-06T00:30:00Z
validation: pending
rollback_target: review-pr-comments-v2
```

이 정도 메타데이터가 있으면 문제가 생겨도 어디로 돌아가야 할지 알 수 있다.

## OpenClaw 같은 구조에서는 더더욱 그렇다

OpenClaw처럼 허브 역할을 하는 시스템에서 자기개선형 스킬이 들어오면 문제는 더 커질 수 있다. 허브는 여러 역할과 도구를 연결하기 때문에, 잘못 생성된 스킬 하나가 여러 흐름에 전파되기 쉽다.

예를 들면 아래처럼 번진다.

- 메시지 분류 규칙이 잘못 학습됨
- 잘못 분류된 요청이 엉뚱한 서브에이전트로 전달됨
- 거기서 생긴 예외 처리가 다시 새 스킬 후보로 저장됨
- 결국 잘못된 라우팅이 강화됨

이런 구조에서는 “배웠다”는 사실보다 **어디에 영향을 주는가**가 더 중요하다.

그래서 허브형 시스템일수록 자기개선 스킬은 아래 원칙이 맞다고 본다.

- 영향 범위가 작은 곳부터 시작한다
- 읽을 수 있는 규칙 파일로 남긴다
- 승인 없는 전역 적용을 피한다
- 검증 로그와 연결한다

## 내가 선호하는 현실적인 자기개선 흐름

완전 자동 학습보다, 나는 아래 같은 반자동 흐름이 훨씬 현실적이라고 본다.

### 단계 1. 패턴 후보 수집
- 반복 실행 로그에서 자주 등장한 절차를 뽑는다
- 성공률, 실패율, 예외 케이스를 함께 붙인다

### 단계 2. 스킬 후보 생성
- 사람이 읽을 수 있는 스킬 초안으로 저장한다
- 적용 범위와 금지 조건을 함께 적는다

### 단계 3. 리뷰와 제한 배포
- 특정 프로젝트나 특정 작업군에만 먼저 적용한다
- 전역 적용은 나중으로 미룬다

### 단계 4. 검증 후 승격
- 재현 가능한 성공 사례가 쌓이면 승인한다
- 실패가 많으면 폐기하거나 분기한다

이 흐름의 장점은 화려하진 않지만, 적어도 사람이 통제권을 잃지 않는다는 점이다.

## 결론

Hermes Agent 같은 아이디어는 충분히 흥미롭다. 에이전트가 경험에서 스킬을 만들고 스스로 개선하는 방향 자체는 분명 앞으로 더 많아질 것이다.

하지만 실전 운영에서 더 중요한 질문은 이거다.

- 스스로 배우는가?
- 보다 먼저,
- **배운 것을 사람이 읽고 승인하고 되돌릴 수 있는가?**

내 기준에서 자기개선형 에이전트가 실용적인 시스템이 되려면 아래가 먼저다.

- 스킬 경계가 선명해야 한다
- 생성 근거가 남아야 한다
- 자동 생성과 자동 적용을 분리해야 한다
- 롤백 가능해야 한다
- 영향 범위를 제한해야 한다

결국 에이전트 자동화에서 중요한 건 “더 많이 배우는 것”만이 아니다. **잘못 배운 것을 시스템 전체 규칙으로 굳히지 않게 만드는 구조**가 더 중요하다.

## 빠른 체크리스트

자기개선형 에이전트나 자동 스킬 생성 시스템을 볼 때는 아래를 먼저 확인해 보면 좋다.

- 새 스킬이 어떤 근거로 생성됐는가?
- 사람이 읽을 수 있는 형식인가?
- 바로 적용되는가, 아니면 후보 상태로 머무는가?
- 실패하면 어떤 버전으로 롤백하는가?
- 전역 규칙으로 퍼지기 전에 영향 범위를 제한하는가?

이 질문에 답이 흐리면, 그 시스템은 똑똑한 자동화라기보다 **디버깅 어려운 규칙 증식기**에 가까울 가능성이 크다.
