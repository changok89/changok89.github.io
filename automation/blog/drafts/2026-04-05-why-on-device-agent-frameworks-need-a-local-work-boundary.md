최근 GeekNews에서 Ailoy라는 온디바이스 AI 에이전트 제작 도구를 봤다. 소개 문구는 꽤 직관적이다. Python과 JavaScript 버전을 제공하고, 머신러닝 전문가가 아니어도 에이전트를 쉽게 만들 수 있으며, 가능하면 외부 API 비용 대신 on-device AI 쪽으로 돌리고 싶다는 문제의식이 깔려 있다.

이런 류의 프로젝트를 볼 때 사람들은 보통 먼저 이렇게 묻는다.

- 성능이 어느 정도인가
- 어떤 모델을 붙일 수 있나
- 진짜 로컬에서 다 돌아가나
- LangChain 같은 기존 툴보다 뭐가 쉬운가

물론 다 중요한 질문이다. 그런데 내 기준으로는 그보다 먼저 봐야 할 것이 있다. **이 도구가 작업 경계를 어떻게 잡게 해 주는가**다.

왜냐하면 온디바이스 에이전트는 모델 하나만으로 성패가 갈리지 않기 때문이다. 실제로는 아래가 더 크게 작용한다.

- 로컬에서 처리할 일과 외부에 맡길 일을 얼마나 명확히 나누는지
- 실패했을 때 어디서 다시 시작할 수 있는지
- 입력/출력 형식이 기존 도구와 얼마나 잘 맞는지
- 비용 절감이 단순한 구호가 아니라 운영 구조로 이어지는지

Ailoy를 보면서도 내가 흥미롭게 본 건 “로컬 AI를 쉽게 붙인다”는 표면적인 메시지보다, **온디바이스 에이전트 프레임워크를 평가할 때 어떤 기준이 실전에서 중요한가**였다.

## 왜 온디바이스 에이전트는 늘 기대보다 빨리 복잡해지나

로컬 실행이라는 말은 처음 들으면 단순하다.

- 내 기기에서 돈 덜 들이고 돌린다
- 민감한 데이터를 외부로 덜 보낸다
- 네트워크가 없어도 어느 정도는 동작한다

그런데 실제 자동화나 에이전트 구조에 넣는 순간 금방 복잡해진다.

### 1. 모델이 돌아가는 것과 작업이 굴러가는 것은 다르다

모델이 로컬에서 응답한다고 해서, 그걸 바로 실전 워크플로우에 넣을 수 있는 건 아니다.

예를 들면 이런 차이가 있다.

- 데모: 채팅 한두 번 해보니 답이 나온다
- 실전: 파일 입력, 툴 호출, 재시도, 예외 처리, 로그 수집, 상태 저장까지 필요하다

많은 로컬 AI 프로젝트가 첫 단계는 잘 보여준다. 하지만 두 번째 단계에서 갑자기 사용자가 직접 glue code를 많이 써야 한다. 그러면 “쉽다”는 장점이 생각보다 빨리 사라진다.

### 2. 비용 절감은 라우팅이 있어야 의미가 있다

“외부 API를 덜 쓰겠다”는 말은 좋다. 그런데 실전에서는 그냥 로컬로 다 돌린다고 비용 최적화가 되지 않는다.

오히려 이런 구조가 더 중요하다.

- 짧고 반복적인 분류/정리/전처리는 로컬
- 긴 글쓰기나 복잡한 추론은 상위 모델
- 민감 정보가 있는 입력은 먼저 로컬 필터링
- 실패 시 상위 모델로 폴백

즉, 핵심은 모델 선택이 아니라 **작업 라우팅**이다.

### 3. 로컬은 성능보다 운영 난이도가 문제일 때가 많다

클라우드 모델은 돈이 들지만 호출 방식이 대체로 일정하다. 반면 로컬 쪽은 환경 차이 때문에 운영 난이도가 급격히 올라간다.

- OS 의존성
- GPU/메모리 편차
- 모델 파일 관리
- 런타임 버전 충돌
- 속도와 정확도의 기기별 차이

그래서 온디바이스 프레임워크는 “로컬에서 된다”보다 **어디까지 추상화해 주는가**가 더 중요하다.

## 내가 온디바이스 에이전트 프레임워크를 볼 때 체크하는 것

Ailoy 하나만의 이야기는 아니다. 앞으로 비슷한 프레임워크를 볼 때도 나는 아래를 먼저 본다.

### 1. 작업 경계가 선명한가

좋은 프레임워크는 사용자가 자연스럽게 이런 질문을 하게 만든다.

- 이 작업은 로컬에 맡겨도 되는가
- 이건 상위 모델로 넘겨야 하는가
- 실패하면 어디서 다시 시도할 것인가
- 이 입력은 외부 전송 전에 가공이 필요한가

반대로 나쁜 프레임워크는 “에이전트를 쉽게 만든다”는 말은 하지만, 결국 모든 결정을 사용자가 ad-hoc으로 하게 만든다.

내가 원하는 건 마법 같은 자동화가 아니다. **경계를 잡기 쉬운 도구**다.

### 2. 입출력 인터페이스가 단순한가

로컬 에이전트 프레임워크가 실제로 살아남으려면, 기존 개발 환경과 연결하기 쉬워야 한다.

예를 들면 아래가 중요하다.

- Python/JavaScript에서 바로 호출 가능한가
- CLI 또는 HTTP 인터페이스가 있는가
- JSON 출력이 안정적인가
- 로깅과 디버깅 포인트가 보이는가

에이전트는 결국 다른 도구와 연결되어야 한다. 인터페이스가 애매하면 프레임워크가 아니라 데모에 가까워진다.

### 3. 로컬 우선이 아니라 로컬 적합성이 보이는가

나는 “무조건 로컬 우선”이라는 문장에는 별로 끌리지 않는다. 그렇게 밀어붙이면 금방 병목이 생긴다.

오히려 더 실용적인 질문은 이쪽이다.

- 어떤 작업이 로컬에 특히 잘 맞나
- 어떤 작업은 애초에 상위 모델로 보내는 게 낫나
- 어디서 폴백할지 기준이 있나

예를 들면 로컬에 잘 맞는 건 보통 아래와 같다.

- 짧은 분류
- 텍스트 전처리
- 민감 정보 마스킹
- 짧은 요약
- 간단한 명령 추천

반면 아래는 아직 상위 모델이 유리한 경우가 많다.

- 긴 문맥 유지
- 장문 초안 작성
- 복잡한 코드 수정
- 다단계 계획 수립

이 구분이 명확할수록 프레임워크는 더 실용적이다.

## 실전에 넣는다면 나는 이렇게 나눌 것 같다

예를 들어 로컬 에이전트 계층을 앱이나 개인 자동화 허브에 붙인다고 해 보자. 나는 대충 이런 구조로 시작할 것 같다.

### 로컬 계층에 맡길 일

- 들어온 요청 1차 분류
- 민감 정보 제거 또는 치환
- 짧은 문서/메모 요약
- 도구 실행 전 파라미터 정리
- 오프라인 상황에서의 최소 응답

### 상위 모델에 맡길 일

- 장문 블로그 초안 작성
- 큰 코드베이스 이해와 수정
- 복잡한 분석 비교
- 여러 제약조건을 동시에 고려한 계획 수립

### 허브가 맡길 일

- 어떤 모델/계층으로 보낼지 라우팅
- 실패 시 재시도 또는 폴백
- 상태 저장
- 결과 전달과 로그 정리

이 구조가 중요한 이유는, 온디바이스 모델의 약점을 숨기기 위해서가 아니다. 오히려 **잘하는 일만 맡겨서 전체 시스템 효율을 높이기 위해서**다.

## 디버깅 관점에서도 경계가 중요하다

에이전트 시스템은 잘될 때보다, 안 될 때 어떻게 보이는지가 더 중요하다.

온디바이스 프레임워크를 붙였는데 결과가 이상하면 보통 아래 셋 중 하나다.

1. 로컬 모델이 원래 처리하기 어려운 작업을 맡김
2. 입력 전처리가 부족함
3. 실패 후 폴백 경로가 없음

이런 상황에서 경계가 선명하면 문제를 빨리 좁힐 수 있다.

예를 들면 체크리스트가 이렇게 나온다.

```text
[ ] 이 작업은 원래 로컬 계층 담당인가?
[ ] 입력이 너무 길거나 복잡하지 않은가?
[ ] 민감 정보 제거/정규화가 먼저 되었는가?
[ ] 실패 시 상위 모델로 넘기는 규칙이 있는가?
[ ] 로그에서 어느 계층이 실패했는지 바로 보이는가?
```

반대로 모든 걸 “에이전트가 알아서” 하게 두면, 디버깅이 거의 불가능해진다. 이건 로컬이든 클라우드든 똑같다.

## Ailoy 같은 도구가 진짜 쉬우려면 필요한 것

Ailoy 소개만 놓고 보면 접근성은 분명 장점이다. Python과 JavaScript를 둘 다 제공하고, 머신러닝 전문가가 아니어도 쓸 수 있다는 메시지도 좋다. 이런 방향은 분명 더 필요하다.

다만 진짜로 “쉬운 에이전트 프레임워크”가 되려면 아래가 같이 있어야 한다고 본다.

- 권장 아키텍처 예시
- 로컬/외부 모델 분담 기준
- 실패 처리와 재시도 예시
- 표준화된 입출력 형식
- 디버깅 가능한 로그 구조

즉, 쉬움은 API 함수 수가 적다는 뜻이 아니다. **운영 중에 덜 헤매게 만드는 설계**에 더 가깝다.

## 내 결론: 온디바이스 에이전트의 핵심은 모델보다 작업 경계다

Ailoy 같은 프로젝트가 계속 나오는 건 좋은 신호다. 이제는 로컬 AI를 단순 데모가 아니라 실제 에이전트 도구로 쓰고 싶어 하는 사람들이 분명히 늘고 있다.

하지만 이 분야에서 진짜 중요한 건 “로컬에서도 된다”는 사실 하나가 아니다.

정리하면 나는 이렇게 본다.

- 온디바이스 에이전트의 가치는 비용 절감 그 자체보다 작업 분담에 있다
- 프레임워크는 모델 성능보다 작업 경계를 쉽게 잡게 해 줘야 한다
- 로컬 계층은 메인 두뇌라기보다 빠른 분류기, 필터, 전처리기, 보조 작업기로 보는 편이 현실적이다
- 실전에서는 라우팅, 폴백, 로그 구조가 결국 품질을 좌우한다

그래서 앞으로 온디바이스 에이전트 프레임워크를 볼 때도 나는 먼저 이 질문부터 할 것 같다.

**이 도구는 모델을 붙이기 쉽게 해 주는가, 아니면 일을 나누기 쉽게 해 주는가?**

실제로 오래 가는 도구는 대체로 두 번째에 더 가깝다.

## 빠른 체크리스트

온디바이스 에이전트 프레임워크를 볼 때 아래 질문에 먼저 답해 보면 좋다.

- 로컬에 맡길 작업과 외부에 맡길 작업이 구분되는가?
- 입력/출력 형식이 기존 앱이나 스크립트와 잘 연결되는가?
- 실패 시 상위 모델 폴백 경로가 있는가?
- 로컬에서 특히 잘 맞는 작업 예시가 있는가?
- 디버깅 로그와 상태 추적이 가능한가?

이 다섯 개가 흐리면, 아무리 “쉽다”고 해도 실제 운영에서는 금방 복잡해질 가능성이 크다.
