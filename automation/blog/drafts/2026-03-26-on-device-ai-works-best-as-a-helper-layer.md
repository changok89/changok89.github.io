# 온디바이스 AI는 메인 모델이 아니라 보조 계층으로 둘 때 더 실용적이다

온디바이스 AI 이야기가 나올 때 자주 보이는 비교가 있다. 

- 클라우드 LLM이 더 똑똑하다
- 온디바이스 모델이 더 빠르고 프라이버시에 유리하다
- 결국 어느 쪽이 이길까

이 비교가 완전히 틀린 건 아니다. 다만 실무에서는 질문 자체가 조금 어긋날 때가 많다. 많은 제품에서 중요한 것은 **메인 모델을 클라우드에서 온디바이스로 통째로 바꿀 수 있는가**가 아니라, **어떤 단계만 기기 안에서 처리하면 전체 경험이 더 좋아지는가**다.

내가 보기에는 온디바이스 AI의 가장 현실적인 위치는 주력 모델을 완전히 대체하는 자리가 아니다. 오히려 **보조 계층(helper layer)** 으로 붙을 때 훨씬 실용적이다.

즉, 무거운 추론과 긴 문맥 처리는 여전히 서버가 담당하더라도, 아래 같은 부분은 기기 안에서 처리하는 식이다.

- 짧은 분류
- 입력 전처리
- 실시간 필터링
- 개인정보 마스킹
- 오프라인 fallback
- 사용자 인터랙션 직전의 초저지연 판단

이 구조는 생각보다 단순하지만, 실제 사용자 경험과 운영 비용 측면에서는 차이를 크게 만든다.

## 왜 “대체” 관점이 자꾸 무리수가 되나

온디바이스 AI가 멋져 보이는 이유는 분명하다.

- 네트워크 없이 동작할 수 있다
- 응답 지연이 짧다
- 개인정보를 서버로 보내지 않을 수 있다
- 호출 비용이 매번 늘어나지 않는다

문제는 여기서 곧바로 “그럼 클라우드 모델 대신 온디바이스로 가면 되겠네”로 점프하면 현실과 부딪힌다는 점이다.

실제 제품에서는 아래 제약이 금방 드러난다.

### 1. 모델 성능보다 배포·운영 제약이 먼저 온다

모델 하나를 기기에 올리는 일은 단순히 파일을 넣는 문제가 아니다.

- 앱 번들 크기 증가
- 초기 다운로드 시간 증가
- 저장 공간 압박
- 모델 업데이트 전략 필요
- 디바이스별 성능 편차
- 발열과 배터리 영향

서버 모델은 중앙에서 교체하면 끝나지만, 온디바이스 모델은 **배포 전략 자체가 기능 설계의 일부**가 된다.

### 2. 긴 문맥과 복잡한 추론은 여전히 무겁다

짧은 intent 분류나 정형화된 작업은 온디바이스에서 꽤 잘 된다. 하지만 긴 문맥 요약, 복잡한 멀티스텝 추론, 광범위한 일반 지식 질의까지 기기 안에서 모두 해결하려고 하면 금방 무거워진다.

실무에서는 “가능하냐”보다 “이 경험을 매일 안정적으로 제공할 수 있냐”가 더 중요하다. 여기서 많은 팀이 온디바이스를 메인 엔진으로 밀어붙이다가 결국 품질 변동, 속도 저하, 디바이스별 격차 문제를 만나게 된다.

### 3. 제품 전체를 보면 혼합 구조가 더 자연스럽다

사용자가 앱에서 겪는 AI 경험은 하나의 추론 호출로 끝나지 않는다.

예를 들면 이런 흐름이다.

1. 사용자의 입력을 받는다
2. 민감 정보가 있는지 확인한다
3. 빠른 분류를 한다
4. 서버 호출이 필요한지 판단한다
5. 필요하면 서버 모델로 넘긴다
6. 결과를 앱 UI에 맞게 후처리한다

이 전체 흐름 중 모든 단계가 초대형 서버 모델을 필요로 하지는 않는다. 오히려 **앞뒤의 작은 판단**을 기기 안에서 처리할수록 전체 경험이 매끄러워진다.

## 온디바이스 AI가 잘 맞는 곳은 “작고 자주 일어나는 판단”이다

온디바이스 AI를 보조 계층으로 두면 특히 빛나는 구간이 있다. 공통점은 대체로 이렇다.

- 입력 길이가 짧다
- 판단이 자주 일어난다
- 왕복 지연이 거슬린다
- 개인정보가 섞일 가능성이 있다
- 실패해도 치명적이지 않은 fallback이 있다

이 기준으로 보면 온디바이스 AI의 좋은 자리들이 꽤 명확해진다.

### 1. 입력 전처리와 정규화

사용자 입력은 늘 지저분하다.

- 음성 인식 후 텍스트 정리
- OCR 결과 정돈
- 명령어 후보 정규화
- 맞춤법/형식 보정
- 필드별 분류

이 단계는 완벽한 대규모 추론보다 **빠르고 반복 가능한 작은 보정**이 중요하다. 그래서 온디바이스로 옮겼을 때 체감이 좋다.

특히 모바일에서는 키 입력 직후나 음성 입력 직후 몇백 밀리초 내에 응답을 주는 것이 중요하기 때문에, 여기서 서버 왕복을 줄이는 의미가 크다.

### 2. 개인정보 마스킹과 로컬 필터링

서버로 보내기 전에 기기 안에서 먼저 걸러내면 좋은 것들이 있다.

- 전화번호
- 이메일
- 주소
- 주민번호/계좌번호 같은 민감 패턴
- 내부 메모의 특정 태그

물론 규칙 기반 정규식만으로도 되는 부분이 많다. 하지만 문맥 기반 탐지가 필요한 경우는 경량 모델이 꽤 유용하다. 이때 온디바이스 모델은 “답변 생성기”보다 **안전 필터** 역할에서 더 설득력이 크다.

### 3. 오프라인 우선 기능

온디바이스 AI가 가장 빛나는 곳 중 하나는 인터넷이 불안정한 상황이다.

예를 들면:

- 현장 점검 앱에서 사진/메모를 즉시 분류
- 여행 중 네트워크가 약한 상태에서 번역 보조
- 오프라인 상태에서 메모 자동 태깅
- 통신이 불안정한 환경에서 음성 명령 해석

이런 기능은 정확도가 서버 모델보다 조금 낮아도 **아예 안 되는 것보다 낫다**. 즉, 온디바이스의 가치는 최고 성능 경쟁보다 **최소 기능 보장**에서 더 크게 드러난다.

### 4. 서버 호출 여부를 결정하는 게이트

이건 실무적으로 특히 중요하다.

모든 요청을 곧바로 서버 LLM으로 보내지 말고, 먼저 기기 안에서 아래를 판별할 수 있다.

- 정말 AI 호출이 필요한가
- FAQ/룰 기반 응답으로 충분한가
- 짧은 로컬 분류로 끝낼 수 있는가
- 민감 정보가 많아 사용자 확인이 먼저 필요한가

이렇게 하면 서버 비용도 줄고, 쓸데없는 호출도 줄고, 사용자 경험도 더 예측 가능해진다. 결국 온디바이스 모델이 메인 답변을 만들지 않아도 **전체 시스템의 효율을 크게 높일 수 있다.**

## 모바일 제품에서는 이 혼합 구조가 특히 현실적이다

웹 서비스보다 모바일 앱에서 온디바이스 AI 보조 계층이 더 설득력 있는 이유가 있다.

### 1. 지연 시간에 대한 민감도가 높다

모바일에서는 1~2초의 대기만으로도 UX가 거칠어 보인다. 특히 사용자가 입력 중이거나, 카메라/음성/스크롤과 상호작용하는 순간에는 더 그렇다.

서버 모델이 최종 결과를 만드는 구조를 유지하더라도, 앞단에서 로컬 분류와 후처리를 넣으면 “즉각 반응하는 느낌”을 만들기 쉽다.

### 2. 개인정보와 권한 이슈가 더 자주 생긴다

모바일 앱은 사진, 연락처, 마이크 입력, 위치 등 민감한 데이터에 가깝다. 이런 상황에서 전부 서버로 보내는 구조는 제품 설명도 어려워지고, 사용자 신뢰도도 낮아진다.

반대로 일부 판단을 기기 안에서 끝내면 아래 같은 문장을 더 자신 있게 말할 수 있다.

- 민감 정보는 기기 안에서 먼저 걸러집니다
- 일부 추천과 분류는 오프라인에서도 동작합니다
- 서버 전송 전 로컬에서 최소화 과정을 거칩니다

이 메시지는 생각보다 강력하다.

### 3. 디바이스 성능 편차를 설계로 흡수할 수 있다

온디바이스 AI를 메인 엔진으로 두면 저사양 기기에서 전체 기능이 무너질 수 있다. 하지만 보조 계층으로 쓰면 전략이 달라진다.

예:

- 고성능 기기: 로컬 분류 + 로컬 후보 생성 + 서버 보강
- 중간 기기: 로컬 분류만 수행
- 저사양 기기: 룰 기반 fallback

즉, **온디바이스 AI를 필수 의존성이 아니라 성능 최적화 계층으로 다루면** 제품 폭이 훨씬 넓어진다.

## 구현할 때 먼저 정해야 할 것들

온디바이스 AI를 붙이려는 팀이 가장 자주 놓치는 것은 모델 선택 자체보다 **역할 정의**다. 아래 질문이 먼저다.

### 1. 로컬 모델의 임무가 무엇인가

이 질문을 모호하게 두면 기능이 금방 비대해진다.

나쁜 정의:

- 서버 없이도 대부분의 AI를 하게 하자

좋은 정의:

- 서버 호출 전 민감 정보 탐지
- 상위 5개 intent 중 하나 분류
- 오프라인 상태에서 임시 태그 생성
- UI 응답 전 300ms 이내의 즉시 피드백 제공

즉, 로컬 모델은 **좁고 명확한 책임**을 가져야 한다.

### 2. 실패했을 때 fallback이 무엇인가

온디바이스 AI는 디바이스 상태에 영향을 받는다.

- 메모리 부족
- 배터리 세이버 모드
- 발열
- 모델 다운로드 실패
- 특정 기기에서의 성능 저하

그래서 “모델이 있으면 더 좋다” 정도의 구조가 안전하다. 실패 시 fallback을 미리 정해 두면 운영 난이도가 훨씬 내려간다.

- 규칙 기반 처리로 축소
- 서버 호출로 우회
- 기능 수준을 낮춘 degraded mode 제공
- 사용자에게 재시도 대신 간단한 대안 제시

### 3. 업데이트 단위를 어떻게 나눌 것인가

온디바이스 모델은 코드처럼 자주 바꾸기 어렵다. 그래서 아래를 미리 정해야 한다.

- 앱 번들 포함 여부
- 첫 실행 후 다운로드 여부
- 모델 버전 호환 정책
- 구버전 모델의 만료 전략
- A/B 테스트 방식

이걸 나중에 생각하면 기능보다 배포가 더 큰 문제가 된다.

### 4. 무엇을 측정할 것인가

온디바이스 AI는 “잘 돌아가는 것 같다”로 운영하면 안 된다.

최소한 아래는 봐야 한다.

- 기기별 평균 추론 시간
- 다운로드 성공률
- 모델 사용률 대비 서버 fallback 비율
- 배터리/메모리 영향
- 기능별 체감 성공률
- 개인정보 필터의 오탐/누락 비율

측정이 없으면 보조 계층이 실제로 도움이 되는지, 그냥 복잡도만 늘리는지 구분할 수 없다.

## 추천하는 구조: 메인은 서버, 경계와 즉시는 로컬

실무에서 비교적 무리 없는 구조를 한 줄로 요약하면 이렇다.

> 메인 모델은 서버에 두고, 경계 판단과 즉시 반응이 필요한 부분은 로컬에 둔다.

조금 더 풀면 아래와 같다.

1. 입력 수집
2. 로컬에서 민감 정보/간단 분류/오프라인 가능성 판단
3. 필요 시 서버 LLM 호출
4. 로컬에서 결과 후처리 및 UI 맞춤 적용
5. 실패 시 degraded fallback 적용

이 구조가 좋은 이유는 단순하다.

- 서버 모델의 강점을 버리지 않는다
- 온디바이스의 지연/프라이버시 장점을 살린다
- 제품 전체를 한 번에 갈아엎지 않아도 된다
- 디바이스 편차를 전략적으로 흡수할 수 있다

## 언제는 온디바이스를 메인으로 둬도 되나

물론 예외는 있다. 아래처럼 문제 범위가 좁고, 실패 비용이 낮고, 오프라인 가치가 매우 큰 경우는 온디바이스가 메인이 될 수 있다.

- 단순 문서/이미지 분류
- 키보드 보조 기능
- 카메라 기반 실시간 감지
- 정해진 범주의 음성 명령
- 개인 메모 태깅과 검색 보조

핵심은 여기서도 “모든 AI 기능을 로컬로”가 아니라, **작업 범위가 좁고 검증 가능할 때**라는 점이다.

## 결론

온디바이스 AI를 이야기할 때 자꾸 클라우드와의 정면 대결 구도로만 보면 판단이 흐려진다. 많은 앱에서 더 좋은 질문은 이것이다.

- 어떤 판단을 기기 안으로 옮기면 사용자 경험이 좋아지는가
- 어떤 데이터는 서버에 가기 전에 로컬에서 걸러야 하는가
- 어떤 기능은 오프라인에서도 최소 수준으로 살아 있어야 하는가
- 어떤 서버 호출은 로컬 게이트 하나만으로 줄일 수 있는가

내 생각에 온디바이스 AI의 강점은 “메인 두뇌를 완전히 대체한다”보다, **시스템의 앞단과 경계면을 더 똑똑하게 만든다**는 데 있다.

그래서 실무적으로는 이렇게 보는 편이 훨씬 낫다.

> 온디바이스 AI는 주연을 빼앗는 기술이라기보다, 전체 제품 경험을 안정적으로 받쳐 주는 보조 계층에 가깝다.

이 관점으로 접근하면 모델 과시보다 제품 설계가 쉬워지고, 성능 기대치도 현실적으로 맞출 수 있다. 그리고 대부분의 팀에게는 그쪽이 훨씬 유용하다.
