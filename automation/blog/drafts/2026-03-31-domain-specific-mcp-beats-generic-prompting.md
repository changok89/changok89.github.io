# 도메인 특화 MCP가 일반 프롬프트보다 실무적인 이유

AI 도구를 쓸 때 많은 사람이 먼저 시도하는 방식은 비슷하다. 챗창을 열고 자연어로 원하는 걸 길게 설명하는 것이다.

- 이 법 조항이 뭔지 알려 줘
- 최근 개정 내역도 같이 보고 싶어
- 관련 판례가 있으면 함께 정리해 줘
- 별표 내용도 표로 보여 줘

이 접근은 초반에는 꽤 잘 먹힌다. 문제는 도메인이 복잡해질수록 정확도와 재현성이 금방 흔들린다는 점이다. 특히 법률, 세무, 사내 규정, 기술 표준처럼 **데이터 구조가 복잡하고 원문 조회 경로가 정해져 있는 영역**에서는 더 그렇다.

최근 GeekNews에 올라온 `Korean Law MCP` 사례가 흥미로웠던 이유도 여기에 있다. 이 프로젝트는 대한민국 법령, 판례, 행정규칙, 자치법규, 해석례 등을 AI가 바로 다룰 수 있는 구조화된 도구 집합으로 감싼다. 핵심은 단순히 "법률 정보를 AI가 읽는다"가 아니다. **복잡한 도메인을 도구 호출 가능한 형태로 바꿔 놓는다**는 점이다.

내가 보기엔 이 차이가 꽤 중요하다. 실무에서 필요한 건 프롬프트를 더 장인처럼 다듬는 일보다, **반복되는 조회와 해석 흐름을 구조화된 인터페이스로 바꾸는 일**에 더 가깝기 때문이다.

## 일반 프롬프트 방식이 복잡한 도메인에서 흔들리는 이유

일반 프롬프트는 유연하다. 질문을 바로 던질 수 있고, 진입장벽도 낮다. 하지만 구조가 복잡한 도메인에서는 아래 문제가 반복된다.

### 1. 검색 단계와 해석 단계가 섞인다

사용자가 "관세법 제38조 알려줘"라고 묻는 순간 실제로 필요한 단계는 하나가 아니다.

1. 정확한 법 이름을 식별한다
2. 해당 법의 식별자나 문서 키를 찾는다
3. 조문 번호를 시스템 형식에 맞게 변환한다
4. 원문을 가져온다
5. 필요하면 관련 판례나 해석례를 이어 붙인다

사람이 보기엔 하나의 질문이지만, 시스템 입장에서는 여러 도구 호출로 분해되는 작업이다. 일반 프롬프트만으로 이걸 안정적으로 처리하려고 하면 검색이 부정확하거나, 최신 원문 확인 없이 설명부터 시작하거나, 단계 중 일부가 생략되기 쉽다.

### 2. 약어와 도메인 표현이 제각각이다

실무 질문은 늘 정식 명칭으로 들어오지 않는다.

- 화관법
- 근기법
- 별표1
- 시행규칙
- 최근 개정본
- 해석례

이런 표현은 도메인에 익숙한 사람에게는 자연스럽지만, 일반 목적 모델에게는 모호성을 만든다. 모델이 추론으로 맞히는 경우도 있지만, 중요한 업무에서는 "대충 맞췄다"가 가장 위험하다.

### 3. 원문 접근이 귀찮고 형식이 불친절하다

복잡한 공공 데이터나 내부 시스템은 대부분 개발자 경험이 좋지 않다.

- 문서 식별자가 따로 있다
- 조문 번호 표현이 사람이 읽는 방식과 다르다
- 첨부 문서가 HWPX나 특수 형식으로 내려온다
- 비교나 이력 조회가 별도 API로 흩어져 있다

이 상황에서 사람이나 모델이 매번 원문에 접근하는 절차를 손으로 다시 조합하면 생산성도 떨어지고 실수도 늘어난다.

### 4. 같은 질문을 다시 했을 때 흐름이 달라진다

일반 프롬프트는 사람에겐 편하지만, 운영 관점에서는 재현성이 약할 수 있다.

- 오늘은 법 조문부터 보여 주고
- 내일은 요약만 먼저 주고
- 어떤 날은 관련 판례를 붙이고
- 어떤 날은 최신 개정 여부를 생략한다

답변 문체가 달라지는 것 자체는 큰 문제가 아닐 수 있다. 하지만 **조회 절차가 매번 달라지는 것**은 실무에서는 꽤 큰 문제다.

## 도메인 특화 MCP는 질문을 "도구 흐름"으로 바꾼다

MCP가 진짜 유용한 이유는 모델에게 더 많은 설명을 붙일 수 있어서가 아니다. 그보다 중요한 건, 모델이 특정 도메인과 상호작용하는 방식을 **자연어 추측**에서 **구조화된 호출**로 바꿔 준다는 점이다.

예를 들어 한국 법령 조회를 위한 도구가 잘 설계되어 있다면 이런 흐름을 기본값으로 만들 수 있다.

- 법 이름 검색
- 조문 원문 조회
- 개정 이력 비교
- 관련 판례 검색
- 해석례 조회
- 별표 문서 추출 및 표 변환

이 구조가 갖춰지면 모델은 "그럴듯한 법률 설명기"가 아니라, **법률 데이터 시스템의 오케스트레이터**에 가까워진다.

## 왜 이게 실무에서 더 낫나

### 1. 자연어 이해와 데이터 조회를 분리할 수 있다

모델은 사용자의 질문 의도를 파악하는 데 강하다. 하지만 정확한 원문 조회와 식별자 매핑은 도구가 하는 편이 낫다.

이렇게 역할을 분리하면 장점이 분명하다.

- 모델은 질문을 해석한다
- MCP 도구는 정확한 데이터를 가져온다
- 마지막 요약은 다시 모델이 담당한다

즉, 강한 부분만 각자 맡게 된다. 이 구조는 법률 말고도 세무, 의료, 사내 정책, 기술 문서 검색 같은 영역에 모두 통한다.

### 2. 반복 작업을 표준 흐름으로 고정할 수 있다

실무에서는 비슷한 질문이 계속 반복된다.

- 특정 법 조항 확인
- 최근 개정 여부 비교
- 관련 판례 모으기
- 일상어를 법률 용어로 바꾸기
- 별표나 시행규칙까지 따라 내려가기

이런 흐름을 프롬프트에만 의존하면 매번 사람이 다시 조율해야 한다. 반대로 MCP 도구가 있으면 자주 쓰는 패턴을 아예 표준 작업처럼 굳힐 수 있다.

예를 들면 이런 식이다.

- `search_law`로 정확한 법 찾기
- `get_law_text`로 조문 원문 가져오기
- `compare_old_new`로 개정 비교
- `search_precedents`나 `search_interpretations`로 관련 해석 확장

여기서 중요한 건 도구 이름 자체보다, **조회 순서를 재사용 가능한 흐름으로 만들었다**는 점이다.

### 3. 긴 설명보다 작은 정확성이 더 중요해진다

법률이나 규정처럼 중요한 영역에서는 유창한 장문보다 작은 정확성이 더 중요하다.

- 정확한 조문을 봤는가
- 최신 버전을 봤는가
- 관련 시행령/시행규칙이 있는가
- 판례나 해석례를 놓치지 않았는가
- 별표나 부칙까지 확인했는가

일반 프롬프트만 쓰면 모델은 자꾸 설명을 먼저 하려 한다. 반면 도메인 특화 MCP가 있으면 먼저 **찾고, 확인하고, 연결하고, 그다음 설명**하는 흐름으로 바꾸기 쉬워진다.

이 차이는 결과 신뢰도에 직접 연결된다.

## `Korean Law MCP` 사례가 보여 주는 것

이번 사례가 인상적인 건 단순히 툴 수가 많아서가 아니다. 오히려 잘 만든 도메인 MCP가 어떤 특징을 가져야 하는지를 비교적 선명하게 보여 준다.

### 1. 검색만 있는 게 아니라 원문 조회와 연결 도구가 있다

좋은 도메인 도구는 검색 결과 링크만 던지고 끝나지 않는다. 실제 업무에서는 검색 다음 단계가 더 중요하다.

- 조문 원문 조회
- 구법/신법 비교
- 조문별 이력 확인
- 관련 판례 연결
- 위임 체계 확인

즉, "찾기"만이 아니라 **업무 맥락에 맞는 후속 단계**가 같이 설계되어 있어야 한다.

### 2. 한국어 도메인 현실을 반영한다

약어 자동 해석, 조문 번호 변환, 법령-시행령-시행규칙의 3단 구조, HWPX 별표 추출 같은 기능은 범용 AI 툴이 잘 챙기지 않는 부분이다. 하지만 실제 한국 법령 실무에서는 이런 부분이 오히려 핵심 마찰 지점이다.

이 말은 곧, 좋은 도메인 MCP는 "AI를 붙였다"보다 **도메인 특유의 짜증나는 부분을 얼마나 잘 흡수했는가**로 평가해야 한다는 뜻이기도 하다.

### 3. 체인 도구는 실무 워크플로를 반영한다

검색 → 조문 조회 → 판례 확인 → 해석례 탐색 같은 흐름은 사용자가 머릿속으로는 자연스럽게 생각하지만, 도구 수준에서는 여러 단계다.

이걸 체인 도구나 복합 워크플로로 묶어 두면 실무 생산성이 확 올라간다. 모델은 각 단계 세부 구현을 매번 고민하지 않고, 더 높은 수준의 판단에 집중할 수 있다.

## 법률 말고도 같은 패턴이 필요한 곳이 많다

이 아이디어는 법률만의 얘기가 아니다. 비슷한 특성을 가진 도메인이라면 거의 다 통한다.

### 1. 사내 정책/보안 규정 검색

- 어떤 정책 문서가 최신인지
- 예외 승인 절차가 무엇인지
- 관련 운영 기준이 어디에 연결되는지

이런 질문도 일반 프롬프트만으로 처리하면 자꾸 오래된 문서를 참조하거나, 상위 규정만 보고 세부 운영 문서를 놓치기 쉽다.

### 2. 세무·회계 업무

- 조항, 예규, 판정례, 서식, 제출 절차가 얽혀 있다
- 최신 개정 여부가 중요하다
- 같은 용어도 문맥에 따라 의미가 달라진다

이런 영역은 자연어 요약보다 원문 조회 흐름이 훨씬 중요하다.

### 3. 내부 기술 운영 문서

- 서비스 런북
- 장애 대응 절차
- 배포 정책
- 인프라 권한 규칙

이 역시 잘 설계된 MCP가 있으면 "문서를 읽고 요약해 주는 AI"에서 한 단계 넘어가, **정확한 문서를 찾고 관련 문서까지 묶어 보여 주는 운영 인터페이스**로 바뀔 수 있다.

## 결국 중요한 건 프롬프트를 버리는 게 아니다

여기서 오해하면 안 되는 점도 있다. 도메인 특화 MCP가 중요하다고 해서 프롬프트가 쓸모없다는 뜻은 아니다.

프롬프트는 여전히 필요하다.

- 사용자의 질문 의도 파악
- 어떤 깊이까지 답할지 결정
- 원문을 어떻게 설명할지 조정
- 초보자용/실무자용 톤 조절

다만 복잡한 도메인일수록 프롬프트의 역할은 "모든 걸 해결하는 본체"가 아니라, **도구를 잘 쓰게 만드는 해석 층**으로 내려오는 편이 더 현실적이다.

즉,

- 프롬프트는 질문을 이해하고
- MCP는 정확한 데이터를 가져오고
- 모델은 마지막 설명과 요약을 맡는 구조

이 조합이 보통 가장 실용적이다.

## 도메인 특화 MCP를 만들 때 체크할 질문

이런 도구를 직접 만들거나 평가할 때는 아래 질문이 유용하다.

1. 검색만 가능한가, 아니면 원문 조회와 후속 연결까지 되는가
2. 약어, 별칭, 도메인 표현을 얼마나 잘 흡수하는가
3. 최신 버전, 이력 비교, 관련 문서 연결이 가능한가
4. 반복 업무를 체인 또는 복합 도구로 묶을 수 있는가
5. 모델이 설명 전에 정확한 조회를 먼저 하도록 유도할 수 있는가
6. 사람이 결과를 검증하기 쉬운 형태로 반환하는가

이 기준으로 보면 좋은 도메인 MCP는 단순한 API 래퍼보다 훨씬 크다. 사실상 **도메인 운영 지식을 도구 인터페이스로 압축한 것**에 가깝다.

## 마무리

AI를 실무에 붙일수록 "프롬프트를 얼마나 잘 쓰느냐"보다 "정확한 도메인 인터페이스를 갖고 있느냐"가 더 중요해지는 순간이 온다.

법률처럼 구조가 복잡한 영역에서는 특히 그렇다. 사용자가 원하는 건 그럴듯한 설명이 아니라, 정확한 조문과 최신 맥락, 관련 판례와 해석 흐름까지 이어지는 **검증 가능한 답변**이기 때문이다.

그래서 도메인 특화 MCP의 가치는 단순히 툴이 많다는 데 있지 않다. 자연어 질문을 **재현 가능한 조회 절차**로 바꾸고, 복잡한 도메인 마찰을 줄이며, AI를 설명기보다 실무 도우미에 가깝게 만든다는 데 있다.

내가 보기엔 앞으로 실무형 AI 도입에서 중요한 경쟁력은 모델 자체보다 이런 도메인 인터페이스 설계에서 더 자주 갈릴 가능성이 크다.
