# AI 에이전트에서 재사용 가능한 Skills를 나누는 방법

에이전트를 조금만 오래 써 보면 금방 부딪히는 문제가 있다. 처음에는 프롬프트를 잘 쓰면 될 것 같았는데, 작업 종류가 늘어나고 반복 실행이 쌓일수록 프롬프트 한 장으로는 감당이 안 되기 시작한다는 점이다.

- 코딩 작업은 어디까지 자동으로 진행할지
- GitHub 작업은 언제 `gh`를 쓰고 언제 브라우저를 써야 할지
- 블로그 초안은 어떤 품질 기준을 통과해야 할지
- 보안 점검은 어떤 항목부터 확인해야 할지
- 외부 쓰기 전에는 언제 멈추고 확인을 받아야 할지

이런 건 단순히 “더 좋은 문장”의 문제가 아니다. 실제로는 **반복되는 작업 경계와 절차를 어떻게 묶어 둘 것인가**의 문제에 가깝다. 그래서 에이전트를 장기적으로 운영할수록 중요한 건 범용 프롬프트를 계속 키우는 일이 아니라, **재사용 가능한 skill을 적절한 크기로 나누는 일**이다.

최근 에이전트 툴링 흐름을 봐도 비슷하다. 사람들은 이제 “프롬프트를 어떻게 잘 쓸까”보다 “반복 작업 단위를 어떻게 패키징할까”에 더 관심을 둔다. 내 기준에서도 이 방향이 훨씬 실무적이다. 좋은 skill 분리는 결과 품질뿐 아니라 유지보수 비용, 안전성, 온보딩 난이도까지 같이 바꾼다.

## 왜 프롬프트 하나로 계속 버티기 어려운가

처음 자동화를 붙일 때는 보통 큰 지시문 하나로 시작한다. 여기에 목표, 형식, 금지사항, 출력 방식, 검토 기준, 예외 처리까지 다 때려 넣는다. 초반에는 이 방식이 빠르다. 하지만 시간이 지나면 거의 항상 같은 문제가 생긴다.

### 1. 한 덩어리 프롬프트가 설정 파일처럼 비대해진다

예를 들어 코딩 작업용 프롬프트를 계속 키우면 이런 내용이 한 파일에 몰린다.

- 어떤 파일부터 읽을지
- 위험한 명령은 언제 멈출지
- 테스트는 어떤 순서로 돌릴지
- Git 작업은 어떻게 할지
- 최종 보고 형식은 어떨지

이렇게 되면 더는 “프롬프트”라기보다 읽기 어려운 운영 문서가 된다. 문제는 이런 문서가 보통 수정에도 약하다는 점이다. 어떤 문장을 빼면 어디에 영향이 가는지 알기 어렵고, 작업 하나를 개선하려다가 다른 작업 품질까지 흔들릴 수 있다.

### 2. 비슷한 작업인데도 세션마다 결과 편차가 커진다

같은 블로그 초안 작업이라도 어느 날은 구조가 잘 나오고, 어느 날은 문제 정의가 빠지고, 어느 날은 검증 단계가 없다. 프롬프트에 같은 내용을 넣는다고 생각해도 실제로는 조금씩 다르게 쓰기 때문이다.

이 편차는 실무에서 꽤 성가시다. 최고점이 낮아서가 아니라 **최저점이 자꾸 흔들리기 때문**이다. 반복 자동화는 한 번 멋진 결과를 내는 것보다, 비슷한 품질을 꾸준히 내는 쪽이 훨씬 중요하다.

### 3. 개선이 누적되지 않고 사람 머릿속 팁으로 남는다

“이번엔 이렇게 시키니까 잘 됐다”는 경험이 생겨도, 그게 다음 실행의 기본값이 되지 않으면 구조적으로는 남는 게 없다. 결국 운영 노하우가 프롬프트 작성자의 기억력에 의존하게 된다.

장기 운영에서 필요한 건 일회성 요령이 아니라, **좋은 절차가 재사용 가능한 기본값으로 굳는 구조**다.

## Skill은 정확히 무엇을 담아야 하나

내가 보는 skill은 단순한 긴 프롬프트가 아니다. 유용한 skill은 보통 아래를 함께 담는다.

- 언제 이 작업을 적용해야 하는지
- 시작 전에 무엇을 읽어야 하는지
- 어떤 도구를 우선 써야 하는지
- 어떤 순서로 진행해야 하는지
- 어떤 경우 사용자 확인이 필요한지
- 결과를 어떤 기준으로 검토해야 하는지

즉, skill의 핵심은 문장 미사여구가 아니라 **작업 흐름의 재사용**이다.

예를 들어 블로그 글쓰기 skill이라면 이런 식의 운영 규칙이 들어간다.

- 기존 블로그 톤과 품질 기준 확인
- 오늘자 초안 존재 여부 확인
- 주제는 기존 사이트 테마와 맞는지 검토
- 초안 작성 후 리뷰 노트 생성
- 억지 주제는 만들지 않기

이런 요소는 그때그때 프롬프트에 손으로 붙일 수도 있다. 하지만 반복될수록 skill로 빼 두는 편이 훨씬 안정적이다.

## Skill 경계를 나누는 현실적인 기준

그렇다고 모든 걸 잘게 쪼개서 skill로 만들면 오히려 복잡해진다. 중요한 건 “무조건 많이”가 아니라 **경계가 분명한 단위로 나누는 것**이다. 내가 유용하다고 느끼는 기준은 아래 네 가지다.

### 1. 작업 목표가 명확히 다른가

가장 쉬운 기준은 목표가 다른지다.

- 코드를 수정하는 작업
- GitHub 이슈/PR을 다루는 작업
- 블로그 초안을 만드는 작업
- 보안 상태를 점검하는 작업
- 모바일 디버깅 문서를 쓰는 작업

이건 서로 필요한 컨텍스트와 검토 기준이 다르다. 한 skill로 우겨 넣기보다 작업 목적별로 나누는 편이 자연스럽다.

### 2. 사용하는 도구 우선순위가 다른가

좋은 skill은 보통 “무슨 도구를 먼저 쓸지”를 함께 포함한다. 이 규칙이 다르면 skill도 나누는 편이 낫다.

예를 들어,

- GitHub 상태 확인은 `gh` CLI가 우선
- 로그인 세션이 필요한 웹 작업은 브라우저가 우선
- 간단한 파일 확인은 read/edit가 우선
- 큰 코딩 작업은 별도 코딩 에이전트가 우선

이런 차이가 있는데도 하나의 범용 skill에 몰아 넣으면, 결국 내부 분기만 복잡해지고 읽기 어려워진다.

### 3. 안전 규칙이 다른가

실무에서는 이 기준이 꽤 중요하다.

- 외부 메시지 발송은 확인 필요
- 삭제성 명령은 사전 동의 필요
- 특정 작업공간에서는 에이전트 스폰 금지
- 운영 서버 변경은 더 보수적으로 진행

안전 제약이 다른 작업을 같은 skill에 넣으면, 규칙이 자꾸 예외문으로 늘어난다. 이런 건 애초에 경계를 분리하는 게 낫다.

### 4. 성공 판정 기준이 다른가

코딩 작업의 성공 기준과 블로그 작업의 성공 기준은 전혀 다르다.

- 코딩: 테스트 통과, 변경 범위, 리스크 정리
- 블로그: 검색 의도 충족, 얇지 않은 내용, 실용 예시, 리뷰 메모
- 점검: 상태 확인, 이상 징후 여부, 후속 조치 제안

완료 기준이 다르면 skill도 분리하는 편이 관리가 쉽다.

## Prompt, Template, Skill을 어디서 갈라야 하나

실제로 가장 많이 헷갈리는 지점이 이 부분이다. 내 기준은 꽤 단순하다.

### 프롬프트에 남길 것

- 이번 작업에만 필요한 맥락
- 특정 이슈나 문서의 세부 요구사항
- 오늘 우선순위
- 일회성 목적 설명

즉, **가변 정보**는 프롬프트에 남긴다.

### 템플릿으로 뺄 것

- 반복되는 출력 형식
- 초안 구조
- 보고서 형식
- 체크리스트 틀

즉, **문서 형식의 재사용**은 템플릿이 맡는다.

### Skill로 뺄 것

- 적용 조건
- 도구 선택 규칙
- 진행 순서
- 안전 제약
- 검토 기준
- 예외 처리 원칙

즉, **운영 규칙의 재사용**은 skill이 맡는다.

이 구분이 있어야 프롬프트가 비대해지지 않고, 수정 위치도 명확해진다.

## 너무 크게 나눈 skill과 너무 잘게 나눈 skill, 둘 다 문제다

Skill 설계에서 흔한 실수는 두 가지다.

### 1. 만능 skill 하나로 다 처리하려는 경우

이 경우 처음에는 편해 보인다. 파일 하나만 보면 되기 때문이다. 하지만 시간이 지나면 아래 문제가 터진다.

- 적용 조건이 지나치게 넓다
- 예외 규칙이 계속 늘어난다
- 도구 우선순위가 충돌한다
- 읽고 수정하기 어려워진다

결국 “재사용”이 아니라 “비대한 운영 문서”가 된다.

### 2. 반대로 너무 잘게 쪼개는 경우

예를 들어 “블로그 제목 뽑기”, “블로그 서론 쓰기”, “블로그 검토하기”를 전부 별도 skill로 만들면, 연결 비용이 늘어난다. 어디서 어떤 skill을 꺼내야 하는지 기억하는 부담도 커진다.

실무에서는 보통 **사람이 하나의 작업이라고 인식하는 덩어리** 정도가 적당하다. 예를 들면 “블로그 초안 작성과 리뷰”, “GitHub PR 상태 확인과 대응”, “호스트 보안 점검”처럼 말이다.

## 내가 추천하는 최소 분리 단위

에이전트를 처음 구조화한다면 아래 정도만 나눠도 체감이 크다.

1. **코딩 작업 skill**
   - 파일 탐색, 수정, 테스트, 커밋 흐름
2. **GitHub 작업 skill**
   - 이슈/PR/CI 확인, 코멘트 대응
3. **글쓰기 skill**
   - 주제 검토, 초안 작성, 리뷰 기준
4. **운영 점검 skill**
   - 상태 확인, 위험 징후 탐지, 권장 조치
5. **연결/설정 진단 skill**
   - 특정 시스템이나 디바이스 연결 문제 해결

이 정도만 분리해도 프롬프트에 몰리던 규칙이 많이 빠진다. 그리고 어느 경계에서 문제가 생기는지도 더 빨리 보인다.

## 좋은 Skill인지 빠르게 판단하는 체크리스트

새 skill을 만들거나 기존 skill을 정리할 때 나는 아래 질문을 해 보면 대부분 감이 온다.

- 이 skill이 적용되는 상황을 한두 문장으로 설명할 수 있는가
- 시작 전에 읽어야 할 정보가 명확한가
- 어떤 도구를 먼저 써야 하는지 분명한가
- 위험한 행동에서 멈추는 조건이 있는가
- 완료 기준이 눈에 보이는가
- 다른 skill과 경계가 겹치지 않는가
- 다음 달의 내가 읽어도 왜 존재하는지 이해할 수 있는가

셋 이상 애매하면 경계를 다시 잡는 편이 낫다.

## 결국 Skill 분리는 운영비를 줄이는 일이다

겉으로 보면 skill 분리는 문서 정리나 프롬프트 정리에 가까워 보일 수 있다. 하지만 실제 효과는 훨씬 운영적이다.

- 결과 편차가 줄어든다
- 온보딩이 쉬워진다
- 안전 규칙이 기본값이 된다
- 개선이 사람 기억이 아니라 시스템에 축적된다
- 어떤 작업이 어디서 실패했는지 추적하기 쉬워진다

특히 에이전트를 혼자만 쓰는 단계에서 팀이나 장기 운영 단계로 넘어갈수록 이 차이가 커진다. 나중에는 프롬프트를 잘 쓰는 사람보다, **반복 업무를 어떤 경계로 구조화했는지**가 전체 품질을 더 많이 좌우한다.

## 정리

재사용 가능한 skill을 잘 나누려면 거창한 프레임워크부터 만들 필요는 없다. 오히려 아래처럼 단순하게 생각하는 편이 실용적이다.

- 이번 일만의 맥락은 프롬프트에 둔다
- 반복되는 형식은 템플릿으로 뺀다
- 절차와 안전 규칙은 skill로 굳힌다
- 목표, 도구, 안전 규칙, 완료 기준이 다르면 skill도 나눈다
- 만능 skill 하나보다 경계가 분명한 몇 개가 낫다

에이전트를 계속 쓸 생각이라면 결국 질문은 “프롬프트를 얼마나 잘 쓸까”가 아니라, **“어떤 반복 지식을 재사용 가능한 경계로 올릴까”**에 가까워진다. 내 경험상 그 경계를 잘 잡은 시스템이 훨씬 오래 간다.
