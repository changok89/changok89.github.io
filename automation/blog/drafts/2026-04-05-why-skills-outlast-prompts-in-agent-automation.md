# 에이전트 자동화에서 프롬프트보다 Skills가 더 오래 가는 이유

AI 에이전트 자동화를 처음 만들 때는 대개 프롬프트에서 출발한다. "이 상황에서는 이렇게 행동해", "이 정보를 보면 저 순서로 처리해" 같은 지시를 시스템 프롬프트나 템플릿에 녹여 넣는 식이다. 시작 단계에서는 이 방식이 빠르고 편하다. 아이디어를 실험하기도 쉽다.

문제는 자동화가 반복 운영 단계로 들어가는 순간부터다. 하루에 한두 번 돌리던 작업이 여러 채널에서 계속 실행되고, 예외 케이스가 쌓이고, 행동 규칙이 늘어나기 시작하면 프롬프트만으로는 금방 관리가 어려워진다.

처음에는 프롬프트가 가장 유연해 보이지만, 시간이 지나면 오히려 변경 비용이 커진다. 반대로 재사용 가능한 **Skills** 단위로 작업을 나누기 시작하면 시스템이 덜 깨지고, 운영 경험도 쌓기 쉬워진다.

내가 보기엔 에이전트 자동화에서 프롬프트는 시작점으로 훌륭하지만, 오래 가는 구조는 대체로 Skills 쪽에서 나온다.

## 프롬프트는 시작하기 쉽지만 경계가 흐리다

프롬프트 중심 설계의 장점은 분명하다.

- 구현 시작 속도가 빠르다
- 자연어로 정책을 바로 적을 수 있다
- 모델의 추론 유연성을 바로 활용할 수 있다
- 작은 실험을 여러 번 돌리기 좋다

그래서 초기 프로토타입에서는 거의 항상 프롬프트가 이긴다. 해야 할 일과 금지할 일을 한 문서에 몰아 쓰고, 예시를 조금 넣으면 꽤 많은 것이 돌아간다.

하지만 이 편리함은 종종 경계가 흐리다는 문제를 동반한다.

예를 들어 아래 같은 규칙이 하나의 프롬프트에 섞이기 시작한다.

- 언제 웹 검색을 해야 하는지
- 어떤 경우 사용자에게 되물어야 하는지
- 특정 채널에서는 어떤 톤으로 말해야 하는지
- 특정 파일을 먼저 읽어야 하는지
- 위험한 작업에서 멈추는 조건이 무엇인지
- 실패 시 재시도할지 종료할지

이게 쌓이면 프롬프트는 점점 "모든 것을 아는 거대한 설명서"가 된다. 문제는 설명이 길어질수록 실제로 어느 규칙이 어떤 상황을 책임지는지 흐려진다는 점이다.

## 운영 단계에서는 재사용 가능한 작업 단위가 필요하다

에이전트 자동화가 실전으로 가면 중요한 건 단순히 한 번 성공하는 것이 아니다.

- 비슷한 일을 반복해도 결과 품질이 유지되는가
- 다른 채널이나 다른 세션에서도 같은 원칙이 적용되는가
- 실패했을 때 어디를 고쳐야 하는지가 분명한가
- 새 작업을 붙일 때 기존 동작을 덜 망가뜨리는가

이 시점부터는 자연스럽게 작업을 더 작은 단위로 나누고 싶어진다. 바로 여기서 Skills 개념이 강해진다.

Skills는 단순히 "좋은 프롬프트 모음"이 아니다. 실제로는 아래에 더 가깝다.

- 특정 작업을 위한 좁은 책임 범위
- 언제 호출해야 하는지에 대한 명확한 트리거
- 작업 순서와 기본 가드레일
- 필요한 입력과 기대 출력의 형태
- 실패 시의 기본 행동
- 참고해야 할 로컬 문서나 스크립트 연결점

이렇게 되면 시스템은 더 이상 모든 상황을 거대한 프롬프트 하나에 우겨 넣지 않아도 된다. 상황에 맞는 작업 단위를 고르고, 그 단위 안에서 필요한 규칙만 불러 쓰면 된다.

## 오래 가는 이유 1: 수정 범위가 작아진다

프롬프트 중심 구조가 오래 버티지 못하는 가장 큰 이유 중 하나는 수정 범위가 너무 넓어지기 때문이다.

예를 들어 GitHub PR 확인 흐름을 조금 개선하고 싶다고 해 보자. 프롬프트 기반 구조에서는 아래 문제가 자주 생긴다.

- GitHub 관련 규칙이 다른 운영 규칙과 섞여 있다
- 어떤 예시를 바꾸면 전혀 다른 상황의 행동도 달라진다
- 의도한 수정인지 회귀인지 파악하기 어렵다

반면 GitHub 확인 전용 Skill이 있다면 수정 범위가 훨씬 작아진다.

- 변경 위치가 분명하다
- 관련 작업에만 영향이 간다
- 실패 시 어디를 되돌려야 하는지 명확하다

소프트웨어에서 함수나 모듈을 나누는 이유와 비슷하다. 자동화도 결국 운영 소프트웨어가 되기 때문에, 수정 반경이 작은 구조가 오래 살아남는다.

## 오래 가는 이유 2: 사람도 읽기 쉬워진다

에이전트 시스템은 모델만 읽는 문서로 운영되지 않는다. 결국 사람도 읽고 고쳐야 한다.

그런데 프롬프트가 너무 커지면 사람이 이해하기 어려워진다.

- 이 규칙이 왜 들어갔는지
- 어느 상황에서 발동하는지
- 지금도 유효한지
- 다른 규칙과 충돌하는지

이걸 파악하는 데 시간이 오래 걸린다.

반대로 잘 나뉜 Skill은 문서 자체가 훨씬 읽기 쉬워진다.

- 목적이 선명하다
- 범위가 좁다
- 입력과 출력이 비교적 예측 가능하다
- 외부 의존성이 보인다

운영자 입장에서는 이 차이가 크다. 시스템이 커질수록 "모델이 알아서 해 준다"보다 "사람이 읽고 유지보수할 수 있다"가 더 중요해진다.

## 오래 가는 이유 3: 실패를 분리해서 다룰 수 있다

에이전트 자동화는 성공 케이스보다 실패 케이스에서 실력이 드러난다. 프롬프트 하나에 모든 행동을 몰아 넣으면 실패도 한 덩어리로 나타난다.

예를 들면 아래 문제가 한꺼번에 섞인다.

- 검색을 잘못했다
- 도구 선택을 잘못했다
- 사용자 의도를 잘못 요약했다
- 금지해야 할 외부 행동을 제안했다
- 로컬 파일 우선 확인 규칙을 놓쳤다

이 상태에서는 원인 분석이 어렵다. 무엇이 망가졌는지보다 "대충 프롬프트를 다시 손보자"로 흐르기 쉽다.

반면 Skill 단위로 나뉘어 있으면 실패를 더 선명하게 볼 수 있다.

- 트리거 선택이 잘못됐는가
- Skill 내부 절차가 부정확한가
- 참고 문서가 낡았는가
- 출력 형식이 다른 단계와 안 맞는가

즉, 실패를 구조적으로 다룰 수 있다. 이게 반복 운영에서 정말 중요하다.

## 오래 가는 이유 4: 도구와 문서를 연결하기 쉬워진다

실전 자동화는 프롬프트만으로 굴러가지 않는다. 파일을 읽고, 로컬 문서를 참고하고, 특정 CLI를 실행하고, 어떤 경우엔 브라우저나 메시징 툴을 써야 한다.

이때 Skills는 좋은 연결 지점이 된다.

예를 들어 특정 작업에서 아래가 필요할 수 있다.

- 먼저 로컬 docs를 읽기
- 특정 파일 경로를 기준으로 작업하기
- 승인 필요한 작업은 사용자 확인을 받기
- 리뷰 후 커밋하기
- 외부 전송은 마지막 단계에서만 허용하기

이런 규칙을 Skill 안에 넣어 두면 단순한 조언이 아니라 **운영 절차**가 된다. 그러면 모델이 상황마다 매번 새로 추론해야 하는 범위가 줄어든다.

결국 좋은 자동화는 똑똑한 자유도가 아니라, 반복 가능한 기본 절차에서 나온다.

## 프롬프트가 쓸모없다는 뜻은 아니다

여기서 중요한 건 프롬프트와 Skills를 대립적으로 볼 필요는 없다는 점이다. 실제로는 둘이 같이 간다.

- 프롬프트는 전역적인 태도와 우선순위를 잡아 준다
- Skills는 구체 작업의 경계와 절차를 정의한다

즉, 프롬프트는 헌법에 가깝고 Skills는 실행 가능한 운영 매뉴얼에 가깝다.

좋은 시스템은 대개 이렇게 나뉜다.

### 프롬프트에 두기 좋은 것
- 기본 톤과 역할
- 안전 원칙
- 도구 사용의 상위 우선순위
- 언제 질문해야 하는지에 대한 전반 규칙

### Skill로 분리하기 좋은 것
- GitHub 작업 흐름
- 건강 점검 루틴
- 블로그 초안 작성 절차
- 특정 서비스 연결 문제 진단
- 리뷰 후 커밋 같은 반복 절차

이렇게 역할을 나누면 전역 규칙은 덜 흔들리고, 개별 작업은 더 빠르게 개선할 수 있다.

## 좋은 Skill은 좁고, 분명하고, 운영 가능해야 한다

그렇다고 아무 작업이나 잘게 쪼개면 오히려 복잡해질 수 있다. Skill은 많다고 좋은 게 아니라, 경계가 좋아야 한다.

내가 선호하는 기준은 이렇다.

### 1. 한 Skill은 한 가지 책임을 가져야 한다

"웹 검색도 하고, GitHub도 보고, 블로그 초안도 쓰고, 마지막에 메시지도 보내기" 같은 덩어리는 너무 넓다. 이렇게 만들면 결국 또 다른 거대 프롬프트가 된다.

### 2. 호출 조건이 분명해야 한다

언제 이 Skill을 써야 하는지 설명하기 어려우면 경계가 흐린 것이다. 좋은 Skill은 설명 한 줄이 나온다.

- PR 상태를 확인할 때
- 보안 하드닝 점검을 할 때
- 날씨 요청을 처리할 때
- 노드 연결 실패를 진단할 때

### 3. 입력과 종료 조건이 보여야 한다

무엇을 받아서 어디까지 처리하는지 보여야 한다. 끝이 없는 Skill은 운영하기 어렵다.

### 4. 외부 행동의 경계가 있어야 한다

읽기 중심인지, 수정 가능한지, 외부 전송까지 포함하는지 선명해야 한다. 특히 메시지 전송, 계정 변경, 공개 쓰기 같은 행동은 더 그렇다.

### 5. 사람이 고칠 수 있어야 한다

Skill 설명이 지나치게 추상적이면 나중에 수정하기 어렵다. 실제로는 사람 유지보수성을 항상 같이 봐야 한다.

## AI 에이전트 생태계가 점점 Skills 쪽으로 가는 이유

최근 에이전트 도구들을 보면 단순히 더 긴 프롬프트를 넣는 방향보다, 재사용 가능한 작업 단위와 운영 절차를 분리하는 방향으로 가고 있다. 이유는 단순하다. 그쪽이 실전 운영에 더 잘 맞기 때문이다.

모델이 좋아질수록 모든 걸 자연어 한 덩어리로 해결할 수 있을 것처럼 보이지만, 실제로는 오히려 그 반대가 드러난다. 모델이 강해질수록 더 많은 일을 맡기고 싶어지고, 그러면 더 명확한 경계와 절차가 필요해진다.

결국 성숙한 자동화는 이런 흐름으로 이동한다.

1. 프롬프트로 빠르게 시작한다
2. 반복되는 작업을 식별한다
3. 실패 패턴을 모은다
4. 안정화할 작업을 Skill로 분리한다
5. Skill 단위로 문서와 절차를 함께 다듬는다

이 과정은 생각보다 소프트웨어 공학에 가깝다. 에이전트 운영도 결국 시스템 설계이기 때문이다.

## 내가 추천하는 실전 기본값

에이전트 자동화를 오래 운영하고 싶다면, 처음부터 거창한 프레임워크를 만들 필요는 없다. 대신 아래 정도를 기본값으로 두는 편이 좋다.

- 전역 프롬프트는 짧고 선명하게 유지하기
- 반복 작업은 Skill로 빼기
- Skill마다 책임 범위와 호출 조건을 적기
- 로컬 문서나 스크립트 연결점을 Skill에 명시하기
- 실패했을 때 어디를 고쳐야 하는지 드러나게 만들기
- 외부 쓰기 행동은 별도 경계로 다루기

이 정도만 해도 시스템이 훨씬 덜 엉킨다.

## 결국 오래 가는 자동화는 설명이 아니라 구조에서 나온다

프롬프트는 강력하다. 특히 처음 뭔가를 움직이게 만들 때는 거의 최고의 도구다. 하지만 자동화가 오래 살아남는 이유는 대개 프롬프트가 길어서가 아니다.

- 작업 경계가 보이고
- 수정 반경이 작고
- 실패 원인을 분리할 수 있고
- 사람도 읽고 유지할 수 있고
- 도구와 문서가 구조적으로 연결되어 있기 때문

그래서 에이전트 자동화에서 프롬프트는 시작점이지만, 장기적으로 더 중요한 건 Skills 같은 재사용 가능한 작업 단위다.

내가 보기엔 앞으로의 에이전트 운영 경쟁력은 "누가 더 그럴듯한 프롬프트를 썼는가"보다 **누가 더 잘 나뉜 작업 구조를 만들었는가**에서 갈릴 가능성이 크다.
