# AI 코딩 에이전트에게 일을 맡길수록 완료 조건을 검증 가능하게 써야 하는 이유

AI 코딩 에이전트를 조금만 오래 써 보면 금방 익숙해지는 장면이 있다. 
“이 버그 좀 고쳐 줘”, “이 기능 추가해 줘”, “테스트도 같이 넣어 줘” 같은 요청을 던지면, 생각보다 빠르게 뭔가 그럴듯한 결과가 나온다. 예전 같으면 반나절은 잡아먹었을 초안이 몇 분 안에 생기기도 한다.

바로 이 지점에서 착각하기 쉽다.

> AI가 코드를 빨리 쓰니까, 이제 중요한 건 더 좋은 프롬프트를 길게 쓰는 일 아닌가?

내 경험상 오래 가는 차이는 프롬프트 길이보다 **완료 조건을 얼마나 검증 가능하게 적느냐**에서 더 크게 벌어진다.

AI가 강해질수록 구현 초안은 빨리 나온다. 하지만 그 결과가 진짜 끝난 것인지, 겉으로만 끝난 것처럼 보이는지, 배포 가능한 수준인지 판단하는 일은 여전히 사람 몫이다. 그래서 “잘 고쳐 줘” 같은 요청보다 아래가 훨씬 중요해진다.

- 무엇이 성공인지
- 무엇은 건드리면 안 되는지
- 어떤 테스트로 확인할지
- 결과를 어떤 형식으로 보고할지
- 남는 리스크는 무엇인지

즉, AI 코딩 시대의 핵심 문장은 멋진 지시문보다 **검증 가능한 완료 조건**에 가깝다.

## 왜 이제 완료 조건이 더 중요해졌나

예전에는 사람이 직접 구현하던 비중이 컸다. 그래서 머릿속에 있는 애매한 요구도 일단 손으로 만지며 다듬을 수 있었다. 구현과 검증이 한 사람 안에서 거의 동시에 일어났다.

하지만 에이전트에게 작업을 넘기면 흐름이 달라진다.

1. 사람이 목표를 준다
2. 에이전트가 구현안을 만든다
3. 사람이 결과를 검토한다
4. 부족하면 다시 지시한다

이 구조에서는 처음부터 완료 기준이 흐리면, 중간 산출물이 아무리 많아도 검토가 어려워진다. 결과가 그럴듯해 보일 뿐, 정말 끝났는지 알기 어렵기 때문이다.

예를 들어 “로그인 버그 고쳐 줘”는 요청처럼 보이지만, 실제로는 거의 아무것도 정의하지 않은 문장에 가깝다.

- 어떤 환경에서 재현되는가
- 어떤 에러가 사라져야 하는가
- 정상 로그인만 고치면 되는가, 자동 로그인도 봐야 하는가
- API 응답 형식 변경은 허용되는가
- 테스트를 어디까지 추가해야 하는가

이런 정보가 빠지면 에이전트는 보통 가장 그럴듯한 해석을 고른다. 문제는 그 해석이 사용자 기대와 다를 수 있다는 점이다.

그래서 AI가 멍청해서가 아니라, **요청이 검증 불가능해서** 반복 루프가 길어지는 경우가 많다.

## "잘 고쳐 줘"가 위험한 이유

막연한 요청이 항상 나쁜 건 아니다. 작은 실험이나 탐색 단계에서는 오히려 빠를 수 있다. 하지만 실제 저장소를 수정하고, 테스트를 돌리고, 커밋 직전까지 가는 작업에서는 얘기가 달라진다.

“잘 고쳐 줘” 계열 요청이 위험한 이유는 크게 네 가지다.

### 1. 작업이 끝났는지 판별하기 어렵다

에이전트는 종종 “수정했습니다”라고 말한다. 하지만 그 말이 의미 있으려면 기준이 있어야 한다.

- 재현 단계 기준으로 에러가 사라졌는가?
- 관련 테스트가 추가됐는가?
- 다른 경로를 깨지 않았는가?
- 요구한 제약을 지켰는가?

이 기준이 없으면 완료 보고는 사실상 자기신고에 가깝다.

### 2. 테스트가 있어도 거짓 안심을 줄 수 있다

AI는 테스트도 잘 만든다. 문제는 테스트가 항상 “올바른 테스트”는 아니라는 점이다.

- 구현 세부사항에 과하게 묶인 테스트
- 실제 버그 재현과 상관없는 테스트
- happy path만 확인하는 테스트
- 원래 깨져야 하는 경우를 놓친 테스트

그래서 단순히 “테스트 추가”만 요구하면 부족하다. **무엇을 증명하는 테스트인지**까지 같이 적어야 한다.

### 3. 리스크가 큰 변경에서 검토 포인트가 흐려진다

인증, 결제, 파일 삭제, 마이그레이션, 배포 스크립트 같은 영역은 같은 한 줄 수정처럼 보여도 위험도가 전혀 다르다. 이런 작업은 완료 조건에 리스크 통제 문장이 들어가야 한다.

예:

- 공개 API 시그니처는 바꾸지 말 것
- 데이터 삭제 로직은 수정하지 말 것
- 기능 변경은 feature flag 뒤에 둘 것
- 실패 시 롤백 방법을 같이 적을 것

이런 제약이 없으면 AI는 기술적으로는 맞지만 운영적으로는 불편한 해법을 선택할 수 있다.

### 4. 결과 보고가 뭉개진다

좋은 완료 조건은 구현 자체뿐 아니라 보고 형식도 바꾼다.

예를 들어 아래 둘은 다르다.

- 수정 후 알려 줘
- 수정한 파일, 테스트 결과, 남은 리스크를 3개 이하 bullet로 요약해 줘

후자는 검토 속도 자체를 올린다. 에이전트 시대에는 구현 속도뿐 아니라 **검토 비용 절감**이 중요하다.

## 검증 가능한 완료 조건은 어떻게 생겼나

완료 조건이라고 해서 거창한 문서가 필요한 건 아니다. 오히려 실무에서는 짧아도 명확한 문장이 더 유용하다.

내 기준으로 좋은 완료 조건은 아래 다섯 가지를 어느 정도 포함한다.

### 1. 성공 상태가 관찰 가능해야 한다

“좋아지게”, “안정적으로”, “깔끔하게” 같은 표현은 인간끼리도 해석이 갈린다. 가능하면 관찰 가능한 상태로 바꾸는 게 낫다.

나쁜 예:

- 로그인 로직을 안정적으로 개선해 줘

좋은 예:

- 잘못된 토큰으로 진입했을 때 앱이 무한 로딩되지 않고 로그인 화면으로 복귀할 것
- 해당 재현 경로를 검증하는 테스트 1개 이상을 추가할 것

### 2. 범위를 같이 적어야 한다

AI는 범위를 안 주면 종종 너무 적게 고치거나 너무 많이 고친다.

예:

- 이번 작업은 `AuthInterceptor`와 로그인 진입 화면까지만 수정할 것
- API 스펙, analytics 이벤트 이름, public interface는 변경하지 말 것

이렇게 적으면 검토자가 “왜 여기까지 바꿨지?”를 덜 묻게 된다.

### 3. 검증 수단이 있어야 한다

완료 조건에는 확인 방법이 포함돼야 한다.

- 어떤 테스트를 돌릴지
- 어떤 시나리오를 재현할지
- 어떤 로그나 화면 변화를 볼지
- 어떤 환경에서 확인할지

예:

- `npm test -- auth-login.spec.ts`가 통과할 것
- 만료 토큰으로 앱 재실행 시 로그인 화면으로 돌아가는지 수동 재현 절차를 적을 것

### 4. 보고 형식을 고정하면 훨씬 좋다

에이전트가 끝낸 뒤 무엇을 보고해야 하는지 미리 정해 두면 리뷰가 쉬워진다.

예:

작업 후 반드시 아래 형식으로 답할 것.

1. 변경한 파일 목록
2. 재현 원인 요약
3. 실행한 테스트와 결과
4. 아직 확인 못 한 리스크

이건 사소해 보이지만 꽤 큰 차이를 만든다. 특히 여러 작업을 병렬로 돌릴 때 더 그렇다.

### 5. 불확실성을 숨기지 못하게 해야 한다

AI가 가장 위험할 때 중 하나는 모르는 부분을 애매하게 덮을 때다. 그래서 완료 조건에 모르면 표시하게 하는 문장을 넣는 편이 낫다.

예:

- 확실하지 않은 추정은 사실처럼 쓰지 말 것
- 테스트하지 못한 경로는 명시할 것
- 권한이 없어서 확인 못 한 부분은 따로 적을 것

이 문장 하나만 있어도 결과 신뢰도가 꽤 올라간다.

## 실무에서 바로 쓰기 좋은 패턴

완료 조건을 잘 쓰는 방법은 결국 자주 쓰는 패턴을 정리해 두는 것이다. 아래는 내가 특히 자주 쓰게 되는 패턴이다.

### 버그 수정 작업

- 재현된 에러 경로를 기준으로 수정할 것
- 관련 없는 리팩터링은 하지 말 것
- 실패하던 경로를 검증하는 테스트를 최소 1개 추가할 것
- 수정 파일, 테스트 결과, 남은 리스크를 함께 보고할 것

### 작은 기능 추가

- 기존 public API는 유지할 것
- 기능 플로우를 깨지 않는 최소 수정으로 구현할 것
- 정상 흐름과 대표 실패 흐름을 각각 확인할 것
- UI 변경 시 사용자 관점에서 달라지는 점을 요약할 것

### 리팩터링

- 동작 변경 없이 구조만 정리할 것
- 변경 전후 테스트 결과를 같이 적을 것
- 성능/가독성 측면에서 얻는 이점을 3줄 이하로 설명할 것
- 동작 변화 가능성이 있는 부분은 별도 표시할 것

### 문서/자동화 작업

- 결과물을 바로 재사용할 수 있는 형식으로 작성할 것
- 추상적 설명보다 명령어, 경로, 예시를 우선할 것
- 사실 확인이 필요한 부분은 TODO 또는 확인 필요로 표시할 것

이런 패턴은 프롬프트를 멋지게 쓰는 것보다 훨씬 재사용 가치가 높다.

## 주니어-시니어 차이는 코드량보다 완료 조건에서 더 드러날 수 있다

AI 도구를 쓰면 겉보기 생산성은 빠르게 평준화될 수 있다. 하지만 완료 조건을 쓰는 방식에서는 오히려 차이가 더 드러난다.

경험이 적을수록 보통 이렇게 요청하기 쉽다.

- 성능 좋아지게 리팩터링해 줘
- 에러 처리 보강해 줘
- 테스트도 추가해 줘

경험이 쌓일수록 보통 이런 식으로 바뀐다.

- `UserSyncService`의 중복 요청만 줄일 것
- 캐시 계층 추가는 이번 범위에서 제외할 것
- 기존 메트릭 이름은 유지할 것
- p95 지연시간 개선은 이번 작업 목표가 아니므로 주장하지 말 것
- 변경 후 영향을 받는 테스트를 실행하고 실패 여부를 적을 것

둘 다 말은 통한다. 하지만 두 번째 요청이 훨씬 덜 위험하고, 결과 검토도 훨씬 빠르다.

결국 AI 시대의 실력 차이는 “얼마나 화려한 프롬프트를 쓰는가”보다 **얼마나 좋은 완료 기준을 설계하는가**에 더 가까워질 수 있다.

## 팀 차원에서도 완료 조건을 재사용해야 한다

개인 사용에서는 머릿속 감각으로도 어느 정도 버틸 수 있다. 하지만 팀으로 가면 완료 조건을 템플릿, 체크리스트, skill 같은 형태로 바깥에 꺼내야 한다.

예를 들어 팀 차원에서 아래 정도는 공통 규칙으로 두는 편이 낫다.

- 위험한 변경에는 반드시 제약 문장을 포함할 것
- 테스트 없는 완료 보고는 완료로 간주하지 않을 것
- 불확실성, 미확인 영역, 권한 부족은 분리해서 적을 것
- 결과 보고 포맷을 통일할 것
- 에이전트가 직접 배포하지 못하는 경계선을 명확히 둘 것

이렇게 해야 검증 기준이 사람 센스가 아니라 운영 체계가 된다.

## 내가 추천하는 최소 템플릿

아주 짧게만 써도 아래 정도면 대부분의 작업 품질이 확 올라간다.

> 목표: 무엇을 고칠지 한 문장으로 쓴다  
> 범위: 어디까지 건드릴지와 건드리지 말 것을 쓴다  
> 완료 조건: 성공 상태를 관찰 가능한 문장으로 쓴다  
> 검증: 실행할 테스트/재현 절차를 쓴다  
> 보고: 수정 파일, 테스트 결과, 남은 리스크를 적게 한다

예시:

> 목표: 만료 토큰 상태에서 앱 재실행 시 무한 로딩되는 로그인 버그 수정  
> 범위: auth 세션 복구 경로만 수정, API 응답 스펙과 analytics 이벤트는 변경 금지  
> 완료 조건: 만료 토큰이면 로그인 화면으로 복귀하고 무한 로딩이 재현되지 않을 것  
> 검증: 관련 단위 테스트 1개 추가, 수동 재현 절차와 결과 기록  
> 보고: 변경 파일, 테스트 결과, 남은 리스크 3개 이하로 요약

이 정도만 써도 에이전트 출력 품질과 리뷰 속도가 꽤 달라진다.

## 마무리

AI 코딩 에이전트가 강해질수록 사람은 더 적게 타이핑할 수 있다. 하지만 그렇다고 일이 단순해지는 건 아니다. 오히려 구현보다 검증의 중요성이 더 커진다.

그래서 내가 보기엔 앞으로 더 중요한 질문은 이것이다.

- 이 작업이 끝났다는 걸 어떻게 확인할 것인가?
- 어떤 제약 아래에서만 이 변경을 허용할 것인가?
- 실패했을 때 무엇이 남는가?
- 에이전트가 무엇을 보고해야 검토가 쉬운가?

좋은 자동화는 화려한 프롬프트에서 시작할 수 있다. 하지만 오래 버티는 자동화는 결국 **검증 가능한 완료 조건** 위에서 굴러간다.

AI에게 더 많은 일을 맡길수록, 사람은 더 좋은 문장을 쓰는 사람보다 **더 좋은 완료 기준을 설계하는 사람**에 가까워질 것이다.
