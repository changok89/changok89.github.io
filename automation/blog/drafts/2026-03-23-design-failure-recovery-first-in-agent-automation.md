# 에이전트 자동화에서 실패 복구를 먼저 설계해야 하는 이유

에이전트 자동화를 처음 붙일 때 사람은 대체로 같은 장면에 먼저 꽂힌다. 
메시지를 보내면 알아서 정리해 주고, 파일을 읽고 수정하고, 웹을 뒤져서 초안을 만들고, 심지어 긴 작업까지 백그라운드에서 처리하는 장면이다. 데모로 보면 꽤 강렬하다. 한 번만 잘 돌아도 "이제 된다"는 감각이 생긴다.

그런데 실제 운영으로 넘어가면 평가는 전혀 다른 기준에서 시작된다. 
사용자는 에이전트가 열 번 중 한 번 멋지게 성공하는 것보다, **멈췄을 때 빨리 드러나고 다시 살아나는지**를 더 민감하게 체감한다.

내 기준으로 에이전트 자동화의 신뢰도는 정답률만으로 결정되지 않는다. 오히려 아래가 더 중요할 때가 많다.

- 실패를 바로 감지할 수 있는가
- 실패 원인을 좁힐 수 있는가
- 자동으로 다시 시도할 수 있는가
- 위험한 상태면 안전하게 멈출 수 있는가
- 이전 정상 상태로 되돌릴 수 있는가

즉, 실전에서는 "얼마나 똑똑한가" 못지않게 **얼마나 복구 가능한가**가 중요하다.

## 데모와 운영은 평가 기준이 다르다

데모 단계에서는 성공 사례가 전부처럼 보인다.

- 한 번 요청했더니 예쁘게 답했다
- 코드 수정이 한 번에 맞았다
- 긴 초안이 그럴듯하게 나왔다
- 툴 호출이 자연스럽게 이어졌다

이 단계에서는 실패가 잘 안 보인다. 설령 실패해도 사람이 바로 옆에서 다시 누르면 그만이기 때문이다.

하지만 운영으로 가면 다른 문제가 튀어나온다.

- 백그라운드 작업이 조용히 멈춤
- 메시지는 받았는데 응답이 안 감
- 응답은 갔는데 실제 작업은 중간에서 끊김
- 브라우저 세션이나 인증 상태가 풀림
- 외부 API 일시 장애로 파이프라인 전체가 멈춤
- 긴 세션이 문맥을 잃고 이상한 반복에 빠짐

이건 모델 성능표에서 잘 안 보이는 문제다. 
그래서 운영 단계에선 정답률보다 **복구 설계**가 훨씬 빨리 병목이 된다.

## 실패는 예외가 아니라 기본값에 가깝다

에이전트 자동화를 설계할 때 가장 위험한 착각 중 하나는 "대부분 잘 돌 테니 실패는 나중에 보자"는 생각이다. 
실제론 그 반대에 가깝다. 구성 요소가 많을수록 실패 지점도 늘어난다.

예를 들어 에이전트 한 번의 작업에도 아래가 걸린다.

- 채널 수신
- 세션 생성 또는 재개
- 툴 권한과 연결 상태
- 파일 시스템 접근
- 외부 네트워크
- 브라우저/셸 실행 상태
- 모델 응답
- 결과 전달

이 중 하나만 어긋나도 작업은 깨진다. 
즉, 실패는 특이한 사고가 아니라 **언젠가 반드시 일어나는 정상 케이스**다.

이 관점이 들어가면 설계 순서도 달라진다. 
기능 구현 다음에 실패 처리를 붙이는 게 아니라, 처음부터 "여기서 깨지면 어떻게 보이고 어떻게 복구할까"를 같이 설계하게 된다.

## 복구 설계를 먼저 하면 얻는 것

### 1. 장애가 조용히 쌓이지 않는다

복구 설계가 없는 시스템은 종종 조용히 죽는다. 
아예 프로세스가 내려가면 그나마 눈에 띄는데, 더 나쁜 건 **반쯤 죽은 상태**다.

- 메시지는 수신되지만 응답이 안 감
- 작업은 시작했는데 완료 이벤트가 안 옴
- 일부 툴만 실패해서 결과 품질이 갑자기 떨어짐
- 특정 채널에서만 응답 누락이 생김

이런 상태는 사람이 한참 뒤에야 눈치채기 쉽다. 
그래서 최소한의 헬스체크와 실패 신호가 필요하다.

### 2. 사람 개입 비용이 줄어든다

복구 설계가 없으면 작은 장애도 결국 사람이 수동으로 확인해야 한다.

- 지금 프로세스가 살아 있는지
- 어디에서 멈췄는지
- 다시 실행하면 되는지
- 같은 입력을 재전송해도 되는지
- 중복 작업이 생기지 않는지

이런 걸 매번 사람이 직접 확인하면 자동화의 이득이 빠르게 줄어든다. 
반대로 재시도, 롤백, 상태 표기가 갖춰져 있으면 사람은 정말 필요한 순간에만 개입하면 된다.

### 3. 신뢰도가 꾸준히 올라간다

사용자는 실패 자체보다도 **실패했을 때의 태도**를 더 오래 기억한다. 
아무 반응 없이 멈추는 시스템은 한두 번만 겪어도 신뢰를 잃는다.

반면 아래처럼 동작하면 체감이 다르다.

- 실패를 바로 알려 준다
- 자동 재시도를 시도한다
- 재시도 후에도 안 되면 안전하게 멈춘다
- 사람이 볼 수 있는 로그나 상태를 남긴다
- 이전 정상 흐름으로 쉽게 복귀할 수 있다

이건 눈에 잘 안 띄는 UX지만, 장기적으로는 기능 추가보다 더 큰 차이를 만든다.

## 에이전트 자동화에서 자주 필요한 복구 패턴

### 1. 재시도는 무조건이 아니라 조건부로

재시도는 가장 먼저 떠오르지만, 무턱대고 붙이면 오히려 문제를 키운다.

재시도가 잘 맞는 경우:

- 일시적인 네트워크 실패
- rate limit
- 외부 API 일시 오류
- 브라우저 로딩 타임아웃
- 잠깐 끊긴 세션 연결

재시도가 위험한 경우:

- 이미 외부 쓰기가 반쯤 끝난 작업
- 중복 메시지 전송 가능성이 큰 작업
- 결제/생성/삭제처럼 멱등성이 약한 작업
- 잘못된 입력 자체가 원인인 작업

핵심은 단순 재시도 횟수보다 **멱등성**과 **부작용 범위**다. 
안전하게 여러 번 시도해도 되는지부터 먼저 판단해야 한다.

### 2. 헬스체크는 살아 있음보다 흐름 점검이 중요하다

프로세스가 떠 있다는 사실만으로는 충분하지 않다. 
에이전트 시스템은 살아 있지만 실질적으로 일하지 못하는 상태가 자주 나온다.

그래서 헬스체크는 보통 두 층으로 보는 편이 좋다.

- **process health**: 프로세스/서비스가 살아 있는가
- **workflow health**: 최근 실제 작업이 정상적으로 처리됐는가

예를 들면 아래가 더 중요할 수 있다.

- 최근 N분 동안 응답이 한 건도 없었는가
- 큐에 작업이 쌓이는데 완료가 안 되는가
- 특정 툴 실패율이 갑자기 치솟았는가
- 브라우저 attach나 인증 관련 실패가 반복되는가

즉, 단순 ping보다 **실제 업무 흐름이 통과하는지**를 보는 편이 훨씬 낫다.

### 3. 롤백 경로는 코드보다 설정에서 더 자주 필요하다

에이전트 장애는 코드 배포 때문만 생기지 않는다. 
오히려 아래가 더 흔하다.

- 프롬프트/지침 변경
- tool wiring 변경
- 권한 정책 변경
- 외부 서비스 토큰 만료
- 모델 교체 또는 세션 정책 변경

그래서 롤백은 배포 시스템만의 문제가 아니다. 
최소한 아래는 되돌릴 수 있어야 한다.

- 이전 설정 파일
- 이전 프롬프트/skill 버전
- 이전 모델/라우팅 선택
- 이전 안정 워크플로

"코드는 그대로인데 왜 갑자기 안 되지?" 같은 상황이 생각보다 많다. 
이럴 때 빨리 돌아갈 기준점을 만들어 둬야 한다.

### 4. 타임아웃은 실패 처리의 시작점이다

에이전트 시스템에서 타임아웃은 귀찮은 옵션이 아니라 **복구 스위치**에 가깝다.

타임아웃이 없으면 작업이 끝났는지 멈췄는지 구분이 안 된다. 
결국 아래가 안 된다.

- 재시도 시점 판단
- 사용자 안내
- stuck session 정리
- 후속 작업 차단

짧은 타임아웃이 항상 좋은 건 아니지만, 적어도 무한 대기는 피해야 한다. 
작업 종류별로 타임아웃을 다르게 두는 편이 현실적이다.

- 짧은 질의응답
- 브라우저 상호작용
- 코드 수정/테스트
- 장기 배치 작업

## 관측 가능성이 없으면 복구도 없다

복구 설계를 하겠다고 해도 로그와 상태가 안 보이면 사실상 할 수 있는 게 적다. 
내 기준으로 최소한 아래 질문에 답할 수 있어야 한다.

- 마지막으로 성공한 작업은 무엇인가
- 마지막으로 실패한 작업은 무엇인가
- 어느 단계에서 실패했는가
- 같은 오류가 반복되고 있는가
- 사람 개입이 필요한 상태인가
- 자동 재시도가 이미 몇 번 실행됐는가

이런 정보가 없으면 장애 대응이 감에 의존하게 된다. 
그러면 결국 원인을 모르니 재시작만 반복하고, 재발 방지도 어려워진다.

작게 시작하더라도 다음 정도는 남기는 편이 좋다.

- 작업 ID 또는 세션 ID
- 시작/완료/실패 시각
- 실패 단계
- 주요 오류 메시지
- 재시도 횟수
- 최종 상태

이 정보만 있어도 "모델이 멍청한 건가?"와 "툴 연결이 깨졌나?"를 훨씬 빨리 구분할 수 있다.

## 내가 선호하는 운영 순서

에이전트 자동화를 새로 만들 때 내 기준으로는 순서가 이렇다.

### 1. 작게 성공하는 흐름 하나 만들기

먼저 끝까지 한 번 통과하는 최소 경로를 만든다. 
메시지 수신, 작업 실행, 결과 전달까지 일단 이어져야 한다.

### 2. 바로 실패 신호를 붙이기

그 다음엔 기능 확장보다 먼저 아래를 붙인다.

- timeout
- error logging
- basic health check
- retry policy
- manual recovery path

여기까지 되어야 비로소 운영 가능한 형태가 된다.

### 3. 그 다음에만 기능을 늘리기

브라우저 자동화, 장기 작업, 다중 채널, 서브에이전트, 스케줄러 같은 건 그 뒤에 붙인다. 
복구 구조 없이 기능만 늘리면 실패 표면만 넓어진다.

## 흔한 실수

### 1. 성공 경로만 데모하고 끝낸다

"잘 될 때"만 보면 시스템이 과대평가된다. 
일부러 네트워크를 끊고, 세션을 죽이고, 토큰을 만료시키고, 응답을 지연시켜 보는 테스트가 필요하다.

### 2. 로그는 나중에 붙이자고 미룬다

대부분의 장애는 처음 겪을 때 원인이 가장 잘 보인다. 
그 시점에 기록이 없으면 다시 재현하느라 시간을 더 쓴다.

### 3. 재시도를 만능 해결책처럼 쓴다

재시도는 일시 장애엔 강하지만, 잘못된 입력이나 논리 오류엔 아무 도움도 안 된다. 
오히려 중복 작업과 비용만 키울 수 있다.

### 4. 사람이 복구하는 절차를 문서화하지 않는다

자동 복구가 완벽할 수는 없다. 
결국 사람이 개입해야 하는 순간이 오는데, 그때 무엇을 보고 어떤 순서로 판단할지 없으면 대응 품질이 들쭉날쭉해진다.

## 마무리

에이전트 자동화는 "한 번 잘 되는가"보다 "망가졌을 때 얼마나 빨리 회복하는가"에서 실전 가치가 드러난다. 
특히 채널, 세션, 툴, 외부 API가 얽힌 구조일수록 실패는 피할 수 없는 기본값에 가깝다.

그래서 내 기준으로는 기능 추가보다 먼저 아래를 챙기는 편이 맞다.

- timeout
- health check
- retry policy
- rollback path
- observable logs and status

정리하면 이렇다.

- 실패는 예외가 아니라 정상 케이스다
- 복구 가능성이 곧 신뢰도다
- 프로세스 생존보다 워크플로 생존을 봐야 한다
- 재시도는 멱등성과 부작용 기준으로 설계해야 한다
- 관측 가능성이 없으면 복구도 없다

에이전트가 똑똑하게 보이는 순간은 많다. 
하지만 오래 살아남는 시스템은 대개 **잘 실패하고, 잘 드러내고, 잘 돌아오는 시스템**이다.
