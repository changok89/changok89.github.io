# 장기 실행 AI 에이전트 앱에서는 모델보다 하네스 설계가 더 중요하다

AI 에이전트 데모를 볼 때는 대개 모델이 먼저 눈에 들어온다. 얼마나 자연스럽게 답하는지, 얼마나 긴 작업을 스스로 이어 가는지, 얼마나 복잡한 요청을 한 번에 처리하는지가 인상 포인트가 되기 쉽다.

그런데 실제로 장시간 돌아가는 에이전트 앱을 운영해 보면 체감이 달라진다. 문제를 일으키는 지점은 종종 모델 자체보다 **그 모델을 둘러싼 실행 하네스(harness)** 다. 다시 말해 세션을 어떻게 만들고, 작업을 어떻게 분리하고, 중간 실패를 어떻게 기록하고, 사용자가 현재 상태를 어떻게 이해하게 할지가 더 중요해진다.

내가 보기에는 장기 실행 AI 앱의 품질은 “가장 똑똑한 모델을 붙였는가”보다 **하네스가 실패를 견디는 구조인가**에 더 크게 좌우된다. 모델은 추론을 담당하지만, 제품을 실제 서비스처럼 움직이게 만드는 것은 하네스다.

## 여기서 말하는 하네스는 무엇인가

이 글에서 하네스는 단순한 래퍼 스크립트를 뜻하지 않는다. 에이전트가 실제 작업을 수행하도록 둘러싼 운영 구조 전체를 말한다.

예를 들면 이런 것들이다.

- 세션 생성과 수명 관리
- 작업 큐와 백그라운드 실행
- 툴 호출 정책과 권한 처리
- 중간 상태 저장과 재개 지점
- 재시도와 타임아웃 정책
- 사용자에게 진행 상황을 보여 주는 방식
- 하위 작업 위임과 결과 회수
- 장애 감지와 복구 루프

이걸 잘 만들면 모델이 다소 흔들려도 서비스는 버틴다. 반대로 이걸 대충 만들면 모델이 아무리 좋아도 사용자 경험은 쉽게 무너진다.

## 왜 장기 실행 앱에서는 하네스의 비중이 급격히 커지나

짧은 질의응답형 챗봇은 한 번 묻고 한 번 답하는 구조라서, 모델 품질이 전면에 드러난다. 하지만 장기 실행 앱은 그보다 훨씬 많은 층이 필요하다.

예를 들면 이런 흐름이다.

1. 사용자가 메신저나 UI에서 작업을 지시한다
2. 시스템이 요청을 정상 접수한다
3. 적절한 세션이나 작업 런타임을 고른다
4. 필요한 파일과 문맥을 준비한다
5. 에이전트가 여러 단계 작업을 수행한다
6. 중간 실패가 나면 재시도하거나 멈춘 지점을 기록한다
7. 결과를 다시 사용자에게 전달한다
8. 필요하면 후속 작업으로 이어 준다

이 흐름에서 모델이 직접 담당하는 것은 일부 단계뿐이다. 실제 운영 리스크는 오히려 모델 바깥에서 더 자주 생긴다.

- 작업은 시작됐는데 완료 알림이 누락됨
- 세션은 살아 있는데 실제 하위 작업은 죽어 있음
- 툴 호출 승인 단계에서 흐름이 멈춤
- 긴 작업 중 문맥이 뒤섞임
- 일부 결과만 생성되고 사용자에게는 실패처럼 보임

이런 문제는 모델 교체만으로 해결되지 않는다. 거의 항상 하네스 문제다.

## 모델이 좋아도 하네스가 약하면 생기는 일

### 1. 성공 경로만 있고 복구 경로가 없다

데모 단계에서는 “잘 되면 멋진” 경로만 있어도 충분해 보인다. 하지만 운영에서는 실패가 전제다.

- 네트워크가 흔들릴 수 있다
- 외부 툴이 느려질 수 있다
- 권한 승인이 지연될 수 있다
- 하위 작업이 일부만 끝날 수 있다
- 메시지 전달이 마지막 단계에서 실패할 수 있다

이때 하네스가 마지막 성공 지점과 실패 원인을 남기지 않으면, 사용자는 그냥 “아무 일도 안 됐다”고 느낀다. 내부적으로 80% 완료된 작업도 사용자 경험 기준으로는 0%가 된다.

### 2. 세션과 작업의 경계가 흐려진다

장기 실행 에이전트에서는 “대화 세션”과 “실제 작업 실행 단위”를 분리해서 봐야 할 때가 많다.

예를 들어 사용자는 하나의 스레드에서 계속 이야기하지만, 내부적으로는 아래가 따로 움직일 수 있다.

- 파일 탐색 작업
- 초안 작성 작업
- 검토 작업
- 커밋 작업
- 배포 확인 작업

이 경계를 분리하지 않으면 중간 실패 후 어디부터 다시 시작해야 하는지 알기 어려워진다. 사용자는 같은 대화를 이어 가고 있는데, 시스템은 이미 다른 상태에 들어가 있을 수 있기 때문이다.

### 3. 사용자 피드백이 늦거나 모호해진다

장기 작업에서 가장 나쁜 경험 중 하나는 “지금 뭘 하고 있는지 전혀 모르겠다”는 상태다.

모델이 아무리 똑똑해도 아래가 안 보이면 불안해진다.

- 요청이 접수됐는지
- 실제 작업이 시작됐는지
- 지금 어느 단계인지
- 멈췄다면 어디서 멈췄는지
- 다시 시도 중인지
- 사람의 개입이 필요한지

즉, 장기 실행 앱에서는 답변 품질만큼 **상태 가시성**이 중요하다.

## 장기 실행 에이전트 앱에서 먼저 설계해야 할 하네스 요소

### 1. 상태 전이를 명확하게 나눠야 한다

최소한 아래 정도는 분리하는 편이 좋다.

- 요청 접수됨
- 실행 준비 중
- 실행 중
- 부분 성공
- 사용자 승인 대기
- 외부 의존성 실패
- 완료
- 전달 실패

이걸 한 덩어리로 “진행 중”이라고만 묶으면 운영이 거의 불가능해진다. 나중에 장애를 볼 때도, 사용자에게 설명할 때도, 자동 복구 규칙을 만들 때도 정보가 부족해진다.

### 2. 재시도는 무조건이 아니라 조건부여야 한다

재시도는 중요하지만, 무조건 반복하면 더 큰 문제를 만든다.

- 같은 실패를 계속 반복해 비용만 늘릴 수 있다
- 중복 메시지를 여러 번 보낼 수 있다
- 이미 완료된 작업을 다시 실행할 수 있다
- 외부 서비스에 과도한 부하를 줄 수 있다

그래서 재시도에는 최소한 아래가 필요하다.

- 재시도 가능한 실패와 아닌 실패의 구분
- 단계별 최대 재시도 횟수
- 지수 백오프나 대기 시간
- 중복 실행 방지 키
- 재개 가능한 마지막 체크포인트

### 3. 긴 작업은 작은 단계로 끊어야 한다

하나의 거대한 프롬프트와 하나의 긴 실행으로 모든 걸 해결하려고 하면 디버깅이 어려워진다. 장기 실행 앱일수록 단계를 작게 쪼개는 편이 낫다.

예를 들면 블로그 자동화도 이렇게 나눌 수 있다.

1. 주제 후보 수집
2. 오늘 쓸 주제 선택
3. 초안 작성
4. 리뷰 노트 생성
5. 수정 반영
6. 게시용 파일 이동
7. 커밋

이렇게 쪼개면 어느 단계에서 실패했는지 바로 보이고, 전체를 처음부터 다시 하지 않아도 된다.

### 4. 사용자에게는 내부 구조보다 행동 가능한 상태를 보여 줘야 한다

운영자는 상세 로그가 필요하지만, 사용자는 다음 행동이 더 중요하다.

좋은 상태 메시지는 대개 이런 식이다.

- 초안 작성은 끝났고 리뷰가 남아 있음
- 외부 검색은 실패했지만 기존 아이디어 백로그 기준으로 초안을 만들 수 있음
- 권한 승인이 필요해서 여기서 멈춰 있음
- 결과 생성은 완료됐지만 전달 재시도 중임

이런 문장은 사용자가 현재 상황을 이해하고 기다릴지, 개입할지, 다시 요청할지를 판단하게 해 준다.

## 실제로 유용한 구조: 모델은 작업자, 하네스는 운영체계

장기 실행 에이전트 앱을 설계할 때 나는 모델을 “전지전능한 자동화 본체”로 보기보다, **하네스 안에서 특정 작업을 수행하는 작업자**로 보는 편이 더 현실적이라고 느낀다.

이 관점으로 보면 설계 우선순위가 바뀐다.

- 어떤 모델을 붙일까
- 가 아니라
- 어떤 작업 단위를 만들까
- 어떤 상태를 저장할까
- 실패 시 어디서 재개할까
- 사용자는 무슨 신호를 받게 할까
- 어느 단계는 사람 승인으로 남길까

이 프레임은 모델 교체에도 강하다. 모델이 바뀌어도 하네스가 안정적이면 시스템 전체를 갈아엎지 않아도 된다.

## 간단한 예시 흐름: 메신저 기반 블로그 초안 자동화

예를 들어 사용자가 메신저로 “오늘 올릴 OpenClaw 관련 글 초안 만들어 줘”라고 보낸다고 해 보자.

이걸 장기 실행 앱처럼 설계하면 흐름은 대략 이렇다.

1. 메시지 수신 후 요청 ID와 시작 시간을 기록한다
2. 오늘 날짜 기준 기존 초안이 있는지 확인한다
3. 없으면 아이디어 백로그와 외부 소스를 읽는다
4. 주제를 하나 고른 뒤 초안을 작성한다
5. 별도 리뷰 단계에서 논지 중복, 얇은 문단, 누락 섹션을 검사한다
6. 통과하면 저장하고 커밋한다
7. 사용자에게 "초안 작성 완료, 리뷰도 끝남"을 보낸다

여기서 중요한 건 모델이 글을 잘 쓰는가만이 아니다. 아래가 더 중요할 수 있다.

- 이미 초안이 있는 날에는 중복 작성 방지
- 외부 검색 실패 시 대체 입력 사용
- 초안과 리뷰를 분리해 품질을 한 번 더 점검
- 저장은 됐지만 메시지 전송이 실패한 경우 재전달 가능
- 커밋 전 상태를 명확히 남겨 다음 단계가 보이게 함

이게 바로 하네스의 역할이다.

## 장기 실행 앱에서 자주 놓치는 질문 다섯 가지

### 1. 이 작업은 어디서 끝났다고 볼 것인가
결과 생성이 끝난 시점인지, 사용자에게 전달된 시점인지, 후속 검토까지 마친 시점인지 정의가 필요하다.

### 2. 실패 후 재실행은 어디부터 할 것인가
처음부터 다시 할지, 마지막 성공 단계부터 이어갈지 정해 두지 않으면 비용과 혼란이 커진다.

### 3. 사용자에게 어떤 중간 상태를 보여 줄 것인가
지나치게 조용하면 불안하고, 너무 시끄러우면 피로하다. 적당한 진행 신호 설계가 필요하다.

### 4. 사람 승인이 필요한 단계는 어디인가
삭제, 배포, 외부 전송, 비용 큰 작업은 자동화보다 승인 지점을 두는 편이 안전하다.

### 5. 모델이 틀려도 시스템이 버티는가
모델이 잘못된 판단을 하더라도 치명적 외부 행동으로 곧장 이어지지 않도록 완충 단계가 있어야 한다.

## 결국 장기 실행 앱의 경쟁력은 "얼마나 똑똑한가"보다 "얼마나 안 무너지는가"에 가깝다

AI 에이전트 앱은 앞으로 더 긴 작업을 맡게 될 가능성이 높다. 코드 수정, 문서 갱신, 운영 점검, 멀티스텝 자동화, 메신저 기반 비동기 작업 같은 것들이다. 이런 영역으로 갈수록 모델 하나의 성능 차이보다 **시스템이 실패를 흡수하고 상태를 설명하며 다시 이어 갈 수 있는지**가 훨씬 중요해진다.

그래서 장기 실행 앱을 만들 때는 모델 선택 전에 먼저 이런 질문을 던지는 편이 낫다.

- 이 작업의 상태 전이는 무엇인가
- 어디까지 자동 재시도할 것인가
- 어떤 단계에서 사람 확인을 받을 것인가
- 멈췄을 때 어디서 재개할 것인가
- 사용자에게 무엇을 알려 줄 것인가

이 질문에 답이 없다면, 좋은 모델을 붙여도 운영은 금방 불안정해진다. 반대로 하네스가 단단하면 모델이 조금 바뀌어도 전체 경험은 오래 버틴다.

## 마무리

장기 실행 AI 에이전트 앱에서 모델은 중요하다. 하지만 실제 서비스 품질을 결정하는 마지막 승부처는 하네스다. 세션 수명, 작업 분리, 상태 기록, 복구 전략, 사용자 피드백이 약하면 똑똑한 모델도 금방 불안정한 제품처럼 보인다.

반대로 하네스를 잘 설계하면 모델은 더 안전하게 쓰이고, 실패는 더 작아지고, 사용자는 시스템을 신뢰하기 쉬워진다. 장기 실행 자동화를 진지하게 만들고 있다면, 다음 최적화 대상은 모델 프롬프트보다 하네스 구조일 가능성이 크다.
