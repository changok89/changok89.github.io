# lipo 란?

`lipo`는 macOS에서 universal binary(여러 아키텍처를 포함한 바이너리)를 다룰 때 사용하는 기본 명령어다. Xcode 개발 환경을 다루다 보면 생각보다 자주 만나게 된다.

대표적인 사용 상황은 아래와 같다.

- 바이너리에 어떤 아키텍처가 들어 있는지 확인할 때
- simulator용 아키텍처를 제거해야 할 때
- device용과 simulator용 라이브러리를 합쳐 universal binary를 만들 때
- 특정 아키텍처만 뽑아내서 디버깅할 때

예전 iOS 라이브러리 배포나 수동 바이너리 정리 작업에서 더 자주 썼지만, 지금도 서드파티 프레임워크나 정적 라이브러리를 직접 관리하는 환경에선 여전히 유용하다.

## universal binary가 뭔가

하나의 바이너리 파일 안에 여러 CPU 아키텍처용 코드가 함께 들어 있는 형태를 말한다.

예를 들면:
- arm64
- x86_64
- armv7 (구형 환경)

이렇게 여러 아키텍처를 묶어두면 환경에 따라 적절한 코드 조각을 선택해서 사용할 수 있다.

## 왜 lipo를 쓰게 되나

실무에서 가장 흔한 이유는 아래 둘이다.

### 1. App Store 업로드 전에 불필요한 아키텍처 제거
라이브러리 안에 simulator용 아키텍처가 남아 있으면 배포 시 문제가 생길 수 있다. 이런 경우 특정 아키텍처를 제거해야 한다.

### 2. device와 simulator 바이너리 병합
서드파티 라이브러리나 수동 빌드 산출물을 배포할 때, 각각 따로 있는 파일을 하나로 합쳐야 하는 경우가 있다.

## 자주 쓰는 lipo 명령

### 1. 아키텍처 목록 확인: `-archs`

입력된 바이너리에 어떤 아키텍처가 들어 있는지 이름만 확인한다.

```bash
lipo -archs MyFramework.framework/MyFramework
```

출력 예:

```bash
arm64 x86_64
```

이 명령은 가장 자주 쓴다. 무언가 제거하거나 합치기 전에 현재 상태를 먼저 보는 용도다.

### 2. 정보 확인: `-info`

아키텍처와 함께 조금 더 읽기 좋은 형식으로 보여준다.

```bash
lipo -info MyFramework.framework/MyFramework
```

출력 예:

```bash
Architectures in the fat file: MyFramework are: arm64 x86_64
```

### 3. universal binary 생성: `-create`

여러 바이너리를 하나로 합친다.

```bash
lipo -create ios/libExample.a simulator/libExample.a -output universal/libExample.a
```

이 방식은 device용, simulator용 산출물이 따로 있을 때 자주 사용한다.

### 4. 특정 아키텍처만 추출: `-extract`

universal binary에서 특정 아키텍처만 분리한다.

```bash
lipo MyFramework.framework/MyFramework -extract arm64 -output MyFramework-arm64
```

### 5. 특정 아키텍처 제거: `-remove`

예를 들어 simulator용 `x86_64`를 제거하고 싶다면 아래처럼 쓸 수 있다.

```bash
lipo MyFramework.framework/MyFramework -remove x86_64 -output MyFramework-stripped
```

### 6. thin 바이너리 만들기: `-thin`

입력한 아키텍처 하나만 포함하는 thin binary를 만든다.

```bash
lipo MyFramework.framework/MyFramework -thin arm64 -output MyFramework-thin
```

`extract`와 비슷한 맥락으로 보이지만, 실제 작업에선 `thin`이 더 직관적으로 느껴질 때가 많다.

## 실무 예시 1: simulator 아키텍처 제거

App Store 업로드 전 또는 배포 패키지 정리 과정에서 simulator 아키텍처가 남아 있으면 문제가 될 수 있다.

예시 흐름:

```bash
lipo -info MyFramework.framework/MyFramework
lipo MyFramework.framework/MyFramework -remove x86_64 -output MyFramework
```

단, 원본 파일을 바로 덮어쓰는 방식은 조심해야 한다. 보통은 임시 파일로 만든 뒤 교체하는 편이 안전하다.

## 실무 예시 2: device/simulator 라이브러리 병합

예전 정적 라이브러리 배포 흐름에선 아래처럼 두 바이너리를 합치는 작업이 흔했다.

```bash
lipo -create build/iphoneos/libSample.a build/iphonesimulator/libSample.a -output build/universal/libSample.a
```

이렇게 만든 파일을 배포하면 소비하는 쪽에서 환경에 따라 적절한 아키텍처를 사용할 수 있다.

## 사용할 때 주의할 점

### 1. 입력 파일이 정말 같은 라이브러리인지 확인
이름만 같고 실제 내용이 다른 바이너리를 합치면 나중에 더 큰 문제를 만든다. 버전, 심볼, 빌드 옵션이 일치하는지 먼저 확인하는 편이 좋다.

### 2. 최신 배포 방식에서는 XCFramework도 고려
최근에는 Apple이 `XCFramework`를 더 권장하는 흐름이라, 새 배포 체계를 설계한다면 `lipo` 기반 universal binary보다 XCFramework 쪽이 더 적절할 수 있다.

즉 `lipo`는 여전히 유용하지만, 모든 경우의 정답은 아니다.

### 3. 원본 덮어쓰기 전에 백업
특정 아키텍처를 제거하는 작업은 되돌리기 번거롭다. 원본 백업 후 작업하는 편이 안전하다.

## 추천하는 확인 순서

작업할 때는 아래 순서가 편하다.

1. `lipo -info` 로 현재 상태 확인
2. 필요한 아키텍처 목록 정리
3. `create`, `remove`, `thin` 중 목적에 맞는 명령 선택
4. 결과 파일에 다시 `lipo -info` 실행해서 검증

## 마무리

`lipo`는 universal binary를 확인하고, 아키텍처를 제거하고, 합치고, 추출할 수 있는 기본 도구다. 오래된 도구처럼 보일 수 있지만, 바이너리를 직접 다뤄야 하는 상황에서는 여전히 유용하다.

한 줄 요약:

- 구조 확인 → `-info`, `-archs`
- 병합 → `-create`
- 제거 → `-remove`
- 특정 아키텍처만 추출 → `-extract`, `-thin`

그리고 새 배포 체계를 설계하는 상황이라면 `lipo`만 보지 말고 `XCFramework`도 함께 검토하는 것이 좋다.
